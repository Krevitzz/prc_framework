# CHARTER PRC 5.3 - ARCHITECTURE NORMALISÉE

# PARTIE I : FONDATIONS

## SECTION 1 : Axiomes du PRC

### A1: Il existe une dissymétrie informationnelle primitive,
notée D, irréductible à toute cause plus fondamentale accessible.

### A2: Il existe un mécanisme d'interaction locale,
noté \Gamma, qui agit sur D.

### A3: Aucune transformation globale et stable \Gamma
ne peut annuler D de manière complète.


## SECTION 2 : Hiérarchie des niveaux


\LEVELS{
  L0 = ontological      # Axiomes immutables (A1, A2, A3)
  L1 = epistemic        # Définitions abstraites (D, \Gamma)
  L2 = theoretical      # Hypothèses testables
  L3 = operative        # Toy models, tests, programmes
  L4 = documentary      # Logs, résultats, analyses
}

**Règle stricte** : Un élément L(n) ne peut référencer que L(n-1) ou inférieur.


## SECTION 3 : Architecture du core

### 3.1 Principe absolu

**Le core PRC = 2 fonctions aveugles (et seulement 2)**

Le core est un moteur d'exécution universel sans aucune logique conditionnelle sur le contenu des données qu'il manipule.

### 3.2 Règles contractuelles du core

#### RÈGLE K1 – Aveuglement total

Le core :
- ne connaît ni la forme de D
- ni sa dimension
- ni son rang
- ni ses symétries
- ni son interprétation

Il manipule des objets opaques (np.ndarray).

#### RÈGLE K2 – Procédure unique et invariante

Le core applique toujours la même procédure :
```
load → apply → record → repeat
```
sans branchement dépendant de D ou \Gamma.

#### RÈGLE K3 – Zéro validation sémantique

Le core ne valide AUCUNE propriété du contenu :
- pas de symétrie
- pas de normalisation
- pas de bornes
- pas de conservation

Toute validation est dans les TM, jamais dans le core

#### RÈGLE K4 – Neutralité structurelle absolue

Le core accepte tout ce que \Gamma accepte.

Il n'impose :
- ni forme
- ni invariants
- ni projections

#### RÈGLE K5 – Séparation stricte exécution / interprétation

Le core ne produit pas des résultats, il produit des données.

Les résultats n'apparaissent que dans les TM via des pipelines d'analyse séparés.

Les TM n'imposent aucune propriété à \Gamma ou D; ils évaluent empiriquement si l'application répétée de \Gamma conduit à une dégénérescence structurelle (trivialité, effondrement de diversité, disparition systématique de symétries pré-existantes) ou à l'émergence de régularités non triviales.

### 3.3 Les 2 fonctions du core

#### Fonction 1 : prepare_state

**Signature** :
```python
def prepare_state(base: np.ndarray,
                  modifiers: Optional[List[Callable]] = None) -> np.ndarray
```

**Responsabilité** :
Composer un état D à partir d'une base et de modificateurs.

**Aveuglement** :
- Ne connaît ni la dimension de `base`
- Ne connaît ni le contenu des `modifiers`
- Ne connaît ni leur interprétation

**Opération** :
Application séquentielle aveugle : `state = modifier_n(...modifier_2(modifier_1(base)))`

---

#### Fonction 2 : run_kernel

**Signature** :
```python
def run_kernel(initial_state: np.ndarray,
               gamma: Callable[[np.ndarray], np.ndarray],
               max_iterations: int = 10000,
               convergence_check: Optional[Callable] = None,
               record_history: bool = False) -> Generator
```

**Responsabilité** :
Appliquer itérativement `state_{n+1} = gamma(state_n)`.

**Aveuglement** :
- Ne connaît ni `gamma` (juste l'applique)
- Ne connaît ni `state` (juste le propage)
- Ne connaît ni leur dimension, structure, ou interprétation

**Opération** :
Génération d'états successifs avec contrôle d'arrêt externe (TM).

**Retour** :
```python
if record_history:
    yield (iteration, state, history)
else:
    yield (iteration, state)
```

### 3.4 Ce qui N'est PAS dans le core

❌ Validation de symétrie  
❌ Validation de bornes  
❌ Validation de positivité  
❌ Enforcement de contraintes  
❌ Classes d'état (State, etc.)  
❌ Classes d'opérateurs (PureSaturation, etc.)  
❌ Utilitaires de création (identity, uniform, etc.)  
❌ Sérialisation  
❌ Analyse  
❌ Visualisation  

**Tout cela va dans des modules séparés.**

### 3.5 Localisation de la validation

```
\VALIDATION_RULES{
  core = none                  # Aucune validation
  encodings = creation_only    # Validation à la création, pas après
  modifiers = passthrough      # Aucune validation
  operators = passthrough      # Aucune validation
  tests = observation_only     # Détecte mais ne corrige jamais
}
```

## SECTION 5 : Nomenclature des états D

### 5.1 Types de D

Le framework distingue plusieurs types d'états D, tous représentés comme `np.ndarray` mais avec des origines différentes.

#### D^(base)

**Définition** : Champ des possibles maximal pour un test donné.

**Exemples** :
- D^(base) symétrique rang 2 : matrice NxN symétrique
- D^(base) asymétrique rang 2 : matrice NxN asymétrique
- D^(base) rang 3 : tenseur NxNxN

**Localisation** : Défini dans `D_encodings/`

**Usage** :
```python
from encodings.rank2_symmetric import create_identity
D_base = create_identity(n_dof=50)
```

#### D^(bruit)

**Définition** : Variations aléatoires appliquées sur D^(base).

**But** : Tester la robustesse/stabilité des patterns émergents.

**Exemples** :
- Bruit Gaussien : `add_gaussian_noise(sigma=0.05)`
- Bruit uniforme : `add_uniform_noise(amplitude=0.01)`

**Localisation** : Défini dans `modifiers/noise.py`

**Usage** :
```python
from modifiers.noise import add_gaussian_noise
D_final = prepare_state(D_base, [add_gaussian_noise(sigma=0.05)])
```

#### D^(topo_x)

**Définition** : Contraintes expérimentales restreignant les DOF, patterns pré-identifiés injectés dans D^(base)

**But** : Simuler l'effet de contraintes physiques.

**Exemples (placeholders)** :
- Double fente : réduction corrélations trans-barrière
- Périodicité : contrainte topologique

**Localisation** : Défini dans `modifiers/constraints.py`

**Note** : Toute la rétro-ingénierie réalité → information reste à faire.

#### D^(plugin)

**Définition** : un ou plusieurs pattern identifiés comme étant associés à une observable (lois, objet, etc).

**But** : Tester la persistance de patterns connus.

**Exemples** :
- Plugin gravitation
- Plugin inertie
- Plugin champ électromagnétique

**Localisation** : `modifiers/plugins/`

**Note** : Nécessite d'abord identification de \Gamma viables et outils de reconnaissance de patterns.

#### D^(domaine)

**Définition** : Compilation pré-remplie de plusieurs plugins.

**Exemples** :
- D^(mécanique classique) = gravitation + inertie + ...
- D^(électromagnétisme) = charge + champ + ...

**Localisation** : `modifiers/domains/`

### 5.2 Composition de D dans un TM

**Principe** : Appel séquentiel de `prepare_state()` avec liste de modifiers.

**Exemple** :
```python
from core.state_preparation import prepare_state
from encodings.rank2_symmetric import create_identity
from modifiers.noise import add_gaussian_noise

D_base = create_identity(50)
D_final = prepare_state(D_base, [
    add_gaussian_noise(sigma=0.05),      # D^(bruit)
])
```

**Note** : Le core (`prepare_state`) ne connaît rien des modifiers, il les applique aveuglément.

## SECTION 6 : Conventions d'identification

### 6.1 Structure ID

**Format** : `TYPE-SUBTYPE-NNN`

**Types principaux** :
- `HYP` = Hypothesis (L2)
- `TM` = Toy Model (L3)
- `TEST` = Test Protocol (L3) 
- `LOG` = Log Entry (L4)
- `RES` = Result Entry (L4)
- `JOUR` = Journal Entry (L4)
- `CON` = Constraint (L2, dérivée)

**Sous-types standards** :
- `GAM` = relatif à \Gamma
- `D` = relatif à D
- `PAT` = relatif aux patterns

### 6.2 Valeurs STATUS

```
\STATUS_VALUES{
  WIP = "Work In Progress"
    - WIP[R0-open] : Exploration rang 0 en cours
    - WIP[R0-closed] : Rang 0 terminé, rang ≥1 non testé
    - WIP[Rn-open] : Exploration rang n en cours
  
  SURVIVES = "Non éliminée par les tests actuels"
    - SURVIVES[R0] : Persiste à rang 0
    - SURVIVES[Rn] : Persiste à rang n
  
  REJECTED = "Hypothèse réfutée ou test échoué"
    - REJECTED[R0] : Éliminée comme mécanisme autonome
    - REJECTED[Rn] : Éliminée à rang n
    - REJECTED[GLOBAL] : Éliminée définitivement tous rangs
}
```

**Note** : Une hypothèse de rang 1 peut contenir une ou plusieurs hypothèse de type WIP[R0-open].

## SECTION 10 : Système de rangs et statuts

### 10.1 Définition des rangs

**Rang 0 (R0)** : Test d'un mécanisme isolé (autonome)

Une hypothèse de rang 0 est testée de manière **isolée**, sans composition avec d'autres mécanismes.

**Caractéristiques** :
- Un seul mécanisme élémentaire
- Aucun couplage
- Aucun héritage d'hypothèses antérieures
- Test d'admissibilité minimale

**Exemple** :
```
HYP-GAM-001 : \Gamma = saturation pure pointwise
→ Testé seul, sans régulation, sans mémoire, sans composition
→ Exploration de rang 0
```
---

**Rang ≥1 (Rn)** : Test de compositions/enrichissements

Une hypothèse de rang n≥1 combine ou enrichit des mécanismes déjà explorés au rang inférieur.

**Caractéristiques** :
- Composition de mécanismes
- Couplages entre opérateurs
- Héritage de contraintes connues
- Test de compatibilité/convergence

**Exemples** :
```
HYP-GAM-002 [R1] : \Gamma = saturation + renforcement hebbian
→ Combine deux mécanismes élémentaires
→ Exploration de rang 1

HYP-GAM-005 [R2] : \Gamma = (saturation + hebbian) + mémoire ordre-1
→ Enrichit une composition de rang 1
→ Exploration de rang 2
```
---

### 10.2 Statuts granulaires normalisés

**Format canonique** :
```
STATUS = [ÉTAT][RANG]
```
**États disponibles** :
- `WIP` 	 : Work In Progress
- `SURVIVES` : Non éliminé par tests
- `REJECTED` : Éliminé

**Qualificateurs de rang** :
- `[R0-open]` : Rang 0 en cours d'exploration
- `[R0-closed]` : Rang 0 exploration terminée
- `[Rn]` : Résultat à rang n spécifique
- `[GLOBAL]` : Rejet définitif tous rangs

---

### 10.3 Hiérarchie stricte des rejets

```
REJECTED[R0] < REJECTED[R1] < ... < REJECTED[GLOBAL]
```

**Règle fondamentale** :
Un rejet à rang n est LOCAL à ce test spécifique.
Seul `REJECTED[GLOBAL]` est définitif tous contextes.

**Exemples** :
```
HYP-001: \Gamma = saturation pure
  REJECTED[R0] → Échoue comme mécanisme isolé
  (Peut être testé dans HYP-010[R1] = saturation + régulation)

HYP-002: \Gamma = diffusion pure  
  REJECTED[R0] → Échoue comme mécanisme isolé
  (Peut être testé dans HYP-010[R1] = diffusion + saturation)

HYP-003: \Gamma = croissance explosive incontrôlable
  REJECTED[GLOBAL] → Violation invariants fondamentaux
  (Ne sera JAMAIS retesté, même en composition)
```

**IMPORTANT** :
- `REJECTED[R0]` ne garantit PAS d'être "bon composant"
- C'est à `HYP[R1]` de CHOISIR quels `REJECTED[R0]` combiner
- Aucune obligation de tester toutes les compositions possibles

---

### 10.4 Statuts WIP détaillés

**WIP[R0-open]**
```
Hypothèse en cours d'exploration à rang 0.
Tests partiels, résultats incomplets.
```

**WIP[R0-closed]**
```
HYP testée à rang 0, résultats non concluants.

Signification:
  - Test R0 terminé sans rejet définitif
  - Aucun verdict clair (ni SURVIVES ni REJECTED)
  - Mise en attente indéfinie
  - Peut être reprise dans HYP[R≥1] futures si pertinent

Ce statut n'implique PAS:
  - Viabilité comme composant
  - Obligation de test à rang supérieur
  - Priorité dans exploration future

Critères typiques WIP[R0-closed]:
  - Réussite partielle des tests (certains PASS, certains FAIL non bloquants)
  - Comportement ambigu nécessitant contexte additionnel
  - Ressources/temps insuffisants pour verdict définitif
```

**Nature du statut WIP** :

WIP n'est PAS un label de qualité.

Ce que WIP signifie:
  ✓ Tests en cours ou suspendus
  ✓ Pas encore de verdict définitif
  ✓ Informations insuffisantes pour conclure

Ce que WIP ne signifie PAS:
  ✗ "Prometteur"
  ✗ "Viable si combiné"
  ✗ "Candidat prioritaire pour R≥1"
  ✗ "Partiellement validé"

**Analogie** :
WIP = dossier médical "en attente de diagnostic"
- Pas "malade mais traitable"
- Pas "probablement sain"
- Juste "données insuffisantes, à reprendre ou archiver"

**Transitions typiques depuis WIP** :
```
WIP[R0-open] → REJECTED[R0]
  (Diagnostic clair: inadmissible)

WIP[R0-open] → SURVIVES[R0]
  (Diagnostic clair: admissible)

WIP[R0-open] → WIP[R0-closed]
  (Diagnostic suspendu: ambigu, ressources épuisées)
```

**Destin probable d'un WIP[R0-closed]** :
- 70% cas: Jamais retesté (oublié, non prioritaire)
- 20% cas: Inclus dans HYP[R1] par opportunité
- 10% cas: Réévalué avec nouveaux critères
---

### 10.5 Statuts SURVIVES détaillés

**SURVIVES[R0]**
```
Hypothèse non éliminée à rang 0.
Peut être testée à rang supérieur ou utilisée en composition.

Critères :
- Aucun échec bloquant à rang 0
- Propriétés structurelles satisfaisantes
- Potentiel de contribution identifié
```

**SURVIVES[Rn], n≥1**
```
Hypothèse non éliminée à rang n.
Continue de persister malgré tests de composition.
```

---

### 10.6 Statuts REJECTED détaillés

**REJECTED[R0]**
```
HYP éliminée dans le contexte de test de rang 0.

Signification:
  - Le test spécifique à R0 a échoué
  - Aucune conclusion sur comportement à R≥1
  - Aucune promotion automatique vers rang supérieur
  - Peut être revisitée dans nouvelle HYP[R1] à discrétion

Ce statut n'est PAS:
  - Une recommandation d'utilisation comme composant
  - Une promesse de test futur en composition
  - Une indication de "potentiel caché"
```

**REJECTED[GLOBAL]**
```
Hypothèse éliminée à tous les rangs.

Critère strict :
Viole des invariants non négociables dans TOUT contexte,
y compris composition et enrichissement.

Exemple :
- Croissance explosive incontrôlable
- Destruction systématique de symétrie
- Incompatibilité mathématique fondamentale
```

---

### 10.7 Transitions de statut

**Autorisées** :
```
WIP[R0-open] → WIP[R0-closed]
WIP[R0-open] → SURVIVES[R0]
WIP[R0-open] → REJECTED[R0]

WIP[R0-closed] → WIP[R1-open]    (promotion verticale)
WIP[R0-closed] → REJECTED[GLOBAL] (rejet global post-analyse)

SURVIVES[Rn] → WIP[Rn+1-open]    (promotion verticale)
SURVIVES[Rn] → REJECTED[Rn+1]    (échec au rang supérieur)

REJECTED[Rn] → REJECTED[GLOBAL]  (après épuisement vertical)
```

**Interdites (violation traçabilité)** :
```
✗ WIP[R0-closed] → WIP[R0-open]      (pas de retour arrière)
✗ REJECTED[Rn] → SURVIVES[Rn]        (pas de réactivation)
✗ REJECTED[GLOBAL] → [anything else] (définitif)
```

---

### 10.8 Indépendance des hypothèses de composition

**Principe** :
Chaque hypothèse de rang ≥1 est AUTONOME et SÉLECTIVE.

**Sélectivité des compositions** :
```
HYP-004[R1]: Test de (HYP-001[R0] + HYP-002[R0])
  → Ne dit RIEN sur HYP-003[R0]

HYP-005[R1]: Test de (HYP-003[R0] seul, avec contexte différent)
  → Indépendant de HYP-004

HYP-006[R1]: Test de (HYP-001[R0] + HYP-003[R0])
  → Nouvelle combinaison, résultats imprévisibles
```

**Non-exhaustivité** :

Il n'existe AUCUNE obligation de:
- Tester toutes les paires (HYP-i, HYP-j)
- Tester tous les triplets (HYP-i, HYP-j, HYP-k)
- Épuiser l'espace combinatoire

**Stratégie de sélection** :
- Intuition théorique (mécanismes complémentaires)
- Contraintes émergentes (CON-XXX suggèrent couplages)
- Économie de tests (priorité aux combinaisons prometteuses)

**Traçabilité** :

Chaque HYP[R1] doit documenter explicitement:
- Quelles HYP[R0] sont incluses (champ `parent_hypotheses`)
- Pourquoi cette combinaison spécifique
- Quelles HYP[R0] sont délibérément exclues

**Exemple** :
```
HYP-004[R1] {
  parent_hypotheses = [HYP-001, HYP-002]
  rationale = "Saturation + diffusion = régulation locale"
  excluded = [HYP-003]
  exclusion_reason = "Mécanisme orthogonal, testé séparément"
}
```

## SECTION 12 : Architecture pipeline

### 12.1 Principe de non-reruns

#### Règle absolue

**Toute modification de scoring, pondération, ou verdict doit être possible SANS réexécuter les runs du kernel.**

#### Conséquences architecturales

1. **Données factuelles et analyses sont SÉPARÉES**
   - Les runs du kernel ne sont exécutés qu'une fois
   - Toute interprétation ultérieure lit les données existantes

2. **Pipeline étape 5 est REJOUABLE**
   - Tests peuvent être ré-appliqués avec nouvelles implémentations
   - Scores peuvent être recalculés avec nouvelles pondérations
   - Verdicts peuvent être régénérés avec nouveaux seuils

3. **Configs sont EXTERNES**
   - Aucune règle de scoring hardcodée dans le code
   - Tous les paramètres dans des fichiers YAML versionnés

#### Interdictions formelles

```
\INTERDICTIONS{
  - Forcer un rerun pour changer une règle de scoring
  - Mélanger données factuelles et interprétations dans une même table
  - Hardcoder des pondérations ou seuils dans le code
  - Créer un pipeline linéaire non rejouable
}
```

---

### 12.2 Architecture batch_runner

#### Règle structurelle

**Un seul point d'entrée : `batch_runner.py`**
# prc_automation/batch_runner.py


Le script unique gère trois modes d'exécution avec **chaînage automatique** des dépendances.

#### Modes d'exécution

##### Mode `--brut`

**Responsabilité** : Collecte des données de runs

```bash
batch_runner --brut --gamma GAM-001
```

**Comportement** :
1. Exécute kernel pour tous (D, modifier, seed) applicables
2. Stocke dans `db_raw` :
   - Executions (métadonnées run)
   - Snapshots (états sauvegardés)
   - Metrics (métriques par itération)
3. **Aucun test appliqué**
4. **Aucune interprétation**

**Vérification préalable** :
- Vérifie si runs existent déjà dans `db_raw`
- Skip les runs existants (append-only)

---

##### Mode `--test`

**Responsabilité** : Application des tests sur données existantes

```bash
batch_runner --test --gamma GAM-001 --config default
```

**Comportement** :
1. **Vérifie que `db_raw` contient les runs nécessaires (càd les datas d'un run ayant mêmes id pour tous les paramètres)**
2. **Si runs manquants : exécute automatiquement `--brut` d'abord**
3. Lit les runs depuis `db_raw`
4. Applique tous les tests applicables (selon matrice d'applicabilité)
5. Calcule scores contextuels avec config spécifiée
6. Stocke dans `db_results` :
   - TestObservations (valeurs brutes tests)
   - TestScores (scores calculés avec config_id)

**Vérification préalable** :
- Si runs absents dans `db_raw` → exécute `--brut` automatiquement
- Vérifie si tests déjà appliqués pour cette config
- Skip les observations existantes (append-only par config_id)

**Rejouabilité** :
- Peut être relancé avec `--config alternative` sans refaire les runs
- Chaque config génère un nouveau jeu de scores avec son config_id

---

##### Mode `--verdict`

**Responsabilité** : Génération des verdicts agrégés

```bash
batch_runner --verdict --gamma GAM-001 --config default --thresholds strict
```

**Comportement** :
1. **Vérifie que `db_results` contient les scores pour config spécifiée**
2. **Si scores manquants : exécute automatiquement `--test` d'abord**
   - Qui lui-même exécutera `--brut` si nécessaire
3. Lit les scores depuis `db_results`
4. Calcule les 3 critères (Majorité, Robustesse, Score global)
5. Applique les seuils du fichier thresholds
6. Stocke dans `db_results` :
   - GammaVerdicts (verdict avec config_id + threshold_id)

**Vérification préalable** :
- Si scores absents pour config → exécute `--test` automatiquement
- Si runs absents → `--test` exécutera `--brut` automatiquement
- Cascade complète jusqu'au premier manquant

**Rejouabilité** :
- Peut être relancé avec `--thresholds alternative` sans refaire scoring
- Chaque combinaison (config, thresholds) génère un verdict distinct

---

##### Mode `--all` (optionnel, raccourci)

**Responsabilité** : Chaîne complète en une commande

```bash
batch_runner --all --gamma GAM-001 --config default --thresholds strict
```

**Équivalent strict de** :
```bash
batch_runner --brut --gamma GAM-001
batch_runner --test --gamma GAM-001 --config default
batch_runner --verdict --gamma GAM-001 --config default --thresholds strict
```

**Usage** : Utile quand on sait que tout est stable. Sinon, modes séparés préférables.

---

#### Chaînage automatique des dépendances

**Graphe de dépendances** :
```
--brut
  ↓ (vérifié automatiquement par --test)
--test (avec config X)
  ↓ (vérifié automatiquement par --verdict)
--verdict (avec config X, thresholds Y)
```

**Exemple concret** :
```bash
# Situation : db vides

# Exécution automatique :

# 2. Exécute --brut d'abord

# 5. Calcule verdict final
```

**Avantage** : Utilisateur n'a jamais besoin de savoir ce qui existe ou pas.

### 12.3 Séparation bases de données

#### Règle structurelle

**Deux bases séparées, rôles distincts et non interchangeables.**

#### db_raw : Données factuelles

**Fichier** : `prc_database/prc_r0_raw.db`

**Contenu** :
```sql
Executions   -- Métadonnées des runs (gamma_id, params, d_base_id, seed, ...)
Snapshots    -- États sauvegardés (iteration, state_blob compressé)
Metrics      -- Métriques brutes par itération (norm, std, min, max, ...)
```

**Propriétés obligatoires** :
- **Append-only** : Jamais de UPDATE ou DELETE
- **Immuable** : Une fois écrit, jamais modifié
- **Complète** : Contient TOUTES les données nécessaires aux analyses futures

**Cas d'usage** :
- Mode `--brut` écrit exclusivement ici
- Modes `--test` et `--verdict` lisent exclusivement d'ici (sauf leurs propres résultats)

---

#### db_results : Analyses et verdicts

**Fichier** : `prc_database/prc_r0_results.db`

**Contenu** :
```sql
TestObservations  -- Résultats bruts des tests (test_name, observation_data JSON)
TestScores        -- Scores calculés (config_id, test_name, score 0-1)
GammaVerdicts     -- Verdicts agrégés (config_id, threshold_id, verdict)
```

**Propriétés obligatoires** :
- **Rejouable** : Peut être supprimée et reconstruite depuis db_raw
- **Versionnée** : Chaque config/threshold génère un jeu de résultats distinct
- **Multiple** : Plusieurs verdicts coexistent pour les mêmes runs

**Cas d'usage** :
- Mode `--test` écrit TestObservations et TestScores
- Mode `--verdict` écrit GammaVerdicts
- Les trois modes peuvent lire d'ici pour vérifier existence
- On avertira un utilisateur voulant refaire un test déjà présent, prévenant que le nouveau résultat remplacera l'existant

---

#### Relation entre les bases

**db_raw** est la **source de vérité unique**.

**db_results** est **dérivée** et peut contenir plusieurs interprétations des mêmes runs.

**Schéma de référence** :
```
db_raw.Executions (exec_id, gamma_id, params, d_base_id, seed, ...)
  ↓ (1-to-N)
db_results.TestObservations (exec_id, test_name, value, ...)
  ↓ (1-to-N)
db_results.TestScores (exec_id, test_name, config_id, score, ...)
  ↓ (agrégation)
db_results.GammaVerdicts (gamma_id, config_id, threshold_id, verdict, ...)
```

**Clé étrangère obligatoire** : `exec_id` dans `db_results` référence `db_raw.Executions.id`

### 12.4 Tests

#### 12.4.1 Principe fondamental

**RÈGLE ABSOLUE** : Les tests OBSERVENT, le scoring INTERPRÈTE.

```
\SEPARATION_STRICTE{
  Tests (L3)    : Mesures factuelles → dict standardisé
  Scoring (L3)  : Interprétations contextuelles → scores 0-1
  Verdicts (L3) : Agrégation scores → décisions finales
}
```

**Interdiction formelle** :
- ❌ Tests ne retournent JAMAIS "PASS" ou "FAIL" (c'est une interprétation)
- ❌ Tests ne jugent JAMAIS si une observation est "bonne" ou "mauvaise"
- ❌ Tests ne contiennent JAMAIS de logique de scoring
- ❌ Utilisation d'objets ou classes pour stocker observations
- ❌ Paramètres hardcodés dans le code (seuils, fenêtres, etc.)

**Autorisations** :
- ✅ Tests retournent "ERROR", "NOT_APPLICABLE" (status technique, pas verdict)
- ✅ Tests retournent statistiques complètes (initial, final, min, max, mean, std)
- ✅ Tests décrivent transitions factuelles ("increasing", "stable", etc.)
- ✅ Format dict pur, JSON-sérialisable
- ✅ Paramètres chargés depuis fichiers YAML externes

---

**Tout test DOIT retourner un dict avec cette structure exacte** : voir/demander template pour l'écriture de tous nouveaux tests: tests/utilities/template_test_xxx_yyy_.py

- `"preserved"` : Propriété maintenue (symétrie, etc.)

- `"destroyed"` : Propriété disparue
- `"absent"` : Propriété jamais présente

##### Distinction ERROR vs NOT_APPLICABLE
ERROR est un état d’exécution, NOT_APPLICABLE un état de validité logique.

**ERROR** : Échec technique lors de l'exécution du test

- Exception Python
- Problème de calcul
- **Ces sorties ne sont jamais transmises au scoring ni aux verdicts**
- Log console uniquement

**NOT_APPLICABLE** : Test invalide pour ce contexte
- Mauvais rang de tenseur
- Shape incompatible
- Propriété absente par construction
- **Ces sorties sont inscrites dans db avec applicable=False**
- Exclues du calcul de score (pas pénalisées)

        # Ne jamais scorer - ces observations ne doivent pas arriver ici
        raise ValueError(f"ERROR status should not reach scoring: {test_name}")
    
    if obs['status'] == 'NOT_APPLICABLE':
        return None  # Exclu du calcul (pas pénalisé)
    
    # Sinon interprète observation...
```

## 12.5 Scoring

### 12.5.1 Principe

**Tests observent, scoring interprète.**

Les tests retournent des **observations pures** (valeurs numériques + transitions).

Le scoring applique un **contexte** et des **pondérations** pour calculer des scores 0-1.

---

### 12.5.2 Structure obligatoire

**Le scoring prend une observation et calcule un score 0-1** :
# tests/utilities/scoring.py


#### 12.5.3 Échelle universelle 0-1

**Tous les scores utilisent 0-1** :
```
0.0 = Destruction/échec/explosion
0.5 = Neutre/conservation
1.0 = Création/amélioration/stabilité
```

**Propriétés conservables** (symétrie, norme, diversité) :
```
(initial, final) → score
(0, 0) → 0.5  # Absent → Absent = Neutre
(1, 0) → 0.0  # Présent → Absent = Destruction
(0, 1) → 1.0  # Absent → Présent = Création
(1, 1) → 0.5  # Présent → Présent = Conservation

Métriques continues :

Normalisation selon règle spécifique (ranges, linear, etc.)

12.5.4 Double pondération
Deux niveaux de poids :

metric_weight : importance métrique DANS son test
test_weight : importance test DANS verdict global

Formule contribution :
score_test = Σ(metric_score × metric_weight) / Σ(metric_weight)
contribution_verdict = score_test × test_weight
```

**Effet amplification** :
- Test weight=2, métrique weight=2 → contribution 4× standard
- Nécessaire pour hiérarchie fine
- Défaut: tous weights = 1.0

**Documentation obligatoire** :
Toute déviation de 1.0 DOIT être justifiée dans THRESHOLDS_SPEC.

---

### 12.5.3 Échelle universelle 0-1

**Tous les tests utilisent l'échelle 0-1** :

```
PROPRIÉTÉS CONSERVABLES (symétrie, norme, diversité, positivité)
├─ 0.0 = Destruction/perte complète
├─ 0.5 = Neutre/conservation
└─ 1.0 = Création/amélioration

MÉTRIQUES CONTINUES (vitesse convergence, rayon spectral)
├─ Normalisation selon distribution observée
├─ 0.0 = borne inférieure du critère
├─ 0.5 = Médiane
└─ 1.0 = borne supérieure du critère
```

**Note** : Les bornes seront définies par test, dans certains cas une convergence rapide permet d'évaluer la stabilité sous perturbation, dans d'autres processus, une convergence lente obtiendra un meilleur score.

**Exemple concret symétrie** :
```
(initial, final) → score
(0, 0) → 0.5  # Absent → Absent = Neutre
(1, 0) → 0.0  # Présent → Absent = Destruction
(0, 1) → 1.0  # Absent → Présent = Création
(1, 1) → 0.5  # Présent → Présent = Conservation (neutre)
```
