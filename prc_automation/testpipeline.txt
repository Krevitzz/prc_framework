explication du pipeline:

---

# 1. SPÉCIFICATION FORMELLE DU PIPELINE — MODE `--test`

## 1.1. Principe général

* Le pipeline de test opère **exclusivement** sur la base de données brute :

  ```
  DB_RAW_PATH = "prc_database/prc_r0_raw.db"
  ```
* Tous les tests :

  * lisent `db_raw`
  * produisent des observations dérivées
* Les tests **ne s’exécutent jamais eux-mêmes**
* Toute exécution est centralisée dans `test_engine.py`

---

## 1.2. Point d’entrée

```text
batch_runner --test
```

Déclenche le pipeline de test décrit ci-dessous.

---

## 1.3. Discovery

### Rôle

* Scanner le répertoire `tests/`
* Charger les fichiers de définition de test dont le nom est conforme ex test_SYM_001.py, et laisse de côté test_STR_002_deprecated.py
* Constituer une liste ordonnée de tests

### Contraintes

* Discovery est **statique**
* Aucune lecture de `db_raw`
* Aucune logique d’applicabilité
* Aucune exécution de test

Sortie :

```python
List[TestDefinition]
```

---

## 1.4. Boucle d’exécution principale

Pour chaque `test_definition`
Pour chaque jeu de paramètres dict data_ID=`{gamma_id, d_base_id, modifier_id, seed, …}`

---

## 1.5. Applicability (`applicability.py`)

### Rôle

* **Décider** si un test est applicable à un jeu de données donné
* La décision repose :

  * sur les métadonnées du test
  * sur le contexte courant
  * sur la disponibilité / compatibilité des données dans `db_raw`

### Comportement

#### Si le test n’est PAS applicable

Le pipeline retourne immédiatement un résultat **standardisé** :

```python
{
    'gamma_id': gamma_id,
    'd_base_id': d_base_id,
    'modifier_id': modifier_id,
    'seed': seed,

    'test_name': TEST_ID,
    'config_params_id': config_params_id,

    'status': 'NOT_APPLICABLE',
    'message': 'NOT_APPLICABLE',

    'statistics': NOT_APPLICABLE,

    'evolution': {
        'transition': NOT_APPLICABLE,
        'trend': NOT_APPLICABLE,
        'trend_coefficient': NOT_APPLICABLE,
    },

    'metadata': {
        'explosion_threshold': NOT_APPLICABLE,
        'stability_tolerance': NOT_APPLICABLE,
        'relative_change': NOT_APPLICABLE,
        'num_iterations': NOT_APPLICABLE,
    }
}
```

* Aucun calcul n’est effectué
* Aucun accès inutile à `db_raw`
* Le test n’est **pas exécuté**

---

## 1.6. Exécution (`test_engine.py`)

### Rôle

`test_engine.py` est **l’unique lieu d’exécution des tests**.

Il est responsable de :

* la lecture de `db_raw`
* la construction des données nécessaires (history, séries, agrégats)
* l’application de la **formule du test**
* le calcul des statistiques
* l’analyse de l’évolution
* la construction du résultat final standardisé

### Comportement

1. Charger les données nécessaires depuis `db_raw`
2. Appliquer la formule déclarée par le test
3. Calculer :

   * `statistics`
   * `evolution`
   * `metadata`
4. Retourner un résultat structuré :

```python
{
    'gamma_id': gamma_id,
    'd_base_id': d_base_id,
    'modifier_id': modifier_id,
    'seed': seed,

    'test_name': TEST_ID,
    'config_params_id': config_params_id,

    'status': 'SUCCESS',
    'message': f"Mesure: {statistics['initial']:.2f} → {statistics['final']:.2f}",

    'statistics': statistics,

    'evolution': {
        'transition': transition,
        'trend': trend,
        'trend_coefficient': trend_coef,
    },

    'metadata': {
        'explosion_threshold': explosion_threshold,
        'stability_tolerance': stability_tolerance,
        'relative_change': relative_change,
        'num_iterations': len(values),
    }
}
```

---

## 1.7. Stockage

* Tous les résultats (SUCCESS ou NOT_APPLICABLE) sont persistés
* Le format est **strictement identique**
* Aucun traitement spécial ultérieur n’est requis

# 2. INTERFACE MINIMALE OFFICIELLE D’UN FICHIER DE TEST

## 2.1. Principe fondamental

Un fichier de test :

* **ne s’exécute pas**
* **ne lit pas `db_raw`**
* **ne décide pas de l’applicabilité**
* **ne construit pas de résultat final**
* **ne contient aucune boucle d’exécution**

Il **déclare uniquement** ce qui est nécessaire pour que le pipeline puisse l’exécuter.

---

## 2.2. Contenu autorisé d’un fichier de test

### 2.2.1. Métadonnées obligatoires

```python
TEST_ID = "UNIV-002"
TEST_CATEGORY = "UNIV"
```

---

### 2.2.2. Spécification d’applicabilité (déclarative)

```python
APPLICABILITY_SPEC = {
    "requires_rank": None,            # None | 2 | 3 | ...
    "allowed_d_types": ["SYM", "ASY"],
    "min_history_length": 10,
}
```

Aucune logique conditionnelle complexe.
Aucune lecture de données.

---

### 2.2.3. Spécification d’exécution

```python
EXECUTION_SPEC = {
    "needs_history": True,
    "statistics": ["min", "max", "mean", "std"],
    "analyze_evolution": True,
}
```

---

### 2.2.4. Formule du test (pure)

```python
def compute(values, params):
    """
    values : données préparées par test_engine
    params : paramètres chargés depuis la config

    Retourne une mesure brute exploitable par test_engine
    """
    return np.var(values)
```

Contraintes absolues :

* fonction pure
* aucun accès DB
* aucune boucle d’exécution globale
* aucun dict de sortie

---

## 2.3. Contenu explicitement interdit

Dans un fichier de test :

* ❌ accès à `db_raw`
* ❌ lecture de fichiers
* ❌ appels à `test_engine`
* ❌ appels à `applicability`
* ❌ construction de résultats finaux
* ❌ logique d’orchestration

---

## 3. Conclusion contractuelle

* Le pipeline est **centralisé**
* Les tests sont **déclaratifs**
* `applicability` décide
* `test_engine` exécute
* `db_raw` est l’unique source de données

        return module

        """
