## Arborescence logique (carte des modules)
- **Batch Runner** (fichier: ./prc_automation/batch_runner.py)
  - **Responsabilité**: Orchestrateur principal du pipeline d'analyse. Point d'entrée qui coordonne les différents modes d'exécution.
  - **Interfaces**: 
    - **Input**: Arguments CLI (--mode, --gamma, --params), données depuis db_raw
    - **Output**: Résultats dans db_results, rapports dans /reports/
  - **Dépend de**: 
    - tests.utilities.discovery (✓ analysé)
    - tests.utilities.applicability (✓ analysé)
    - tests.utilities.test_engine (✓ analysé)
    - tests.utilities.verdict_engine (commenté mais référencé)
  - **Appelé par**: Ligne de commande (main)

- **Database Raw Schema** (fichier: ./prc_automation/prc_database/schema_raw.sql)
  - **Responsabilité**: Schéma de la base de données SQLite pour le stockage des données brutes (exécutions, snapshots, métriques). Structure optimisée pour la performance avec compression et indexation.
  - **Interfaces**:
    - **Input**: Résultats d'exécutions de simulations (états, métriques, paramètres)
    - **Output**: Données structurées prêtes pour l'analyse par le pipeline de tests
  - **Dépend de**: SQLite (système de base de données)
  - **Appelé par**: batch_runner.py pour charger les données brutes

- **Database Results Schema** (fichier: ./prc_automation/prc_database/schema_results.sql)
  - **Responsabilité**: Schéma de la base de données SQLite pour le stockage des résultats d'analyse des tests (observations, statistiques, évolutions). Conforme au standard Charter 5.5.
  - **Interfaces**:
    - **Input**: Observations générées par test_engine.py
    - **Output**: Données structurées pour analyse de verdict et génération de rapports
  - **Dépend de**: SQLite (système de base de données)
  - **Appelé par**: batch_runner.py pour stocker les résultats des tests

- **Test Discovery** (fichier: ./tests/utilities/discovery.py)
  - **Responsabilité**: Découverte et validation automatique des modules de test selon l'architecture Charter 5.5.
  - **Interfaces**:
    - **Input**: Système de fichiers (recherche de fichiers test_*.py)
    - **Output**: Dictionnaire {test_id: module} des tests valides
  - **Dépend de**: Aucun module interne (seulement bibliothèques standard)
  - **Appelé par**: batch_runner.py (ligne 95)

- **Test Applicability** (fichier: ./tests/utilities/applicability.py)
  - **Responsabilité**: Vérification de l'applicabilité des tests aux exécutions spécifiques basée sur les métadonnées.
  - **Interfaces**:
    - **Input**: Module test, métadonnées de l'exécution (state_shape, d_encoding_id, etc.)
    - **Output**: Tuple (applicable: bool, raison: str)
  - **Dépend de**: Aucun module interne (seulement bibliothèques standard)
  - **Appelé par**: batch_runner.py (ligne 99)

- **Test Engine** (fichier: ./tests/utilities/test_engine.py)

    - **Output**: Dict observation standardisé (statistiques, évolution, métadonnées)
  - **Dépend de**:

## Flux de données global (schéma conceptuel)
```
[Arguments CLI] → batch_runner.py → [Mode sélectionné]
    ↓
[Mode brut] → db_raw (lecture/vérification via schema_raw.sql)
[Mode test] → db_raw → Tests (découverts par discovery.py) → Filtrage (applicability.py) → 
                Exécution (test_engine.py) → Résolution fonctions (registry_manager.py) → 
                Base Registry (classes concrètes) + Post-processors + Chargement config (config_loader.py) → db_results (via schema_results.sql)
[Mode verdict] → db_results → Analyse → Rapports
[Mode all] → Séquence complète
```

### ./prc_automation/prc_database/schema_raw.sql
- **Rôle**: Schéma SQL pour la base de données des données brutes (db_raw). Stocke les exécutions complètes avec leurs paramètres, snapshots compressés, et métriques pré-calculées pour accès rapide.
- **Tables principales**:
  1. **Executions**: Table principale stockant toutes les métadonnées d'exécution
     - Champs clés: `id`, `run_id` (unique), `gamma_id`, `d_encoding_id`
     - Paramètres markoviens: `alpha`, `beta`, `gamma_param`, `omega`
     - Paramètres non-markoviens: `memory_weight`, `window_size`, `epsilon`
     - Paramètres stochastiques: `sigma`, `lambda_param`
     - Paramètres structurels: `eta`, `subspace_dim`
     - Configuration: `modifier_id`, `seed`, `max_iterations`, `snapshot_interval`
     - Résultats: `status`, `error_message`, `final_iteration`, `execution_time_seconds`
     - Contrainte UNIQUE complexe garantissant l'unicité des combinaisons de paramètres
  2. **Snapshots**: Stocke les états compressés (pickle + gzip) à différents itérations
     - Champs: `exec_id`, `iteration`, `state_blob` (BLOB compressé)
     - Métriques rapides pré-calculées: `norm_frobenius`, `norm_spectral`, `min_value`, `max_value`, `mean_value`, `std_value`
  3. **Metrics**: Stocke les métriques par itération (sans l'état complet)
     - Champs: `exec_id`, `iteration`
     - Métriques: normes (`norm_frobenius`, `norm_spectral`, `norm_max`)
     - Statistiques: `min_value`, `max_value`, `mean_value`, `std_value`
     - Distances et propriétés: `distance_to_previous`, `asymmetry_norm`
- **Points critiques**:
  1. **Compression efficace**: `state_blob` utilise pickle + gzip pour réduire la taille de stockage.
  2. **Optimisation requêtes**: Métriques rapides pré-calculées dans Snapshots pour éviter la décompression.
  3. **Intégrité données**: Contrainte UNIQUE complexe dans Executions empêche les doublons.
  4. **Performance**: Index multiples sur les champs de requête fréquents.
  5. **Flexibilité paramètres**: Tous les paramètres possibles inclus (NULL si non applicable).
- **Questions ouvertes**:
  1. Comment sont générés les `run_id` (ex: "GAM-001_beta2.0_SYM-001_M0_s1")?
  2. Quel est le volume typique de données stockées dans `state_blob`?
  3. Comment sont choisis les intervalles de snapshot (`snapshot_interval`)?

### ./prc_automation/prc_database/schema_results.sql
- **Rôle**: Schéma SQL pour la base de données des résultats (db_results). Stocke les observations de tests conformément au standard Charter 5.5, optimisé pour l'extraction rapide et l'analyse de verdict.
- **Table principale**:
  1. **TestObservations**: Table unique stockant toutes les observations de tests
     - Champs de traçabilité: `observation_id` (PK), `exec_id`, `test_name`, `test_category`
     - Configuration: `params_config_id` (identifiant de configuration utilisée)
     - Applicabilité: `applicable` (booléen), `status` (SUCCESS/ERROR/NOT_APPLICABLE), `message`
     - Statistiques extraites: `stat_initial`, `stat_final`, `stat_min`, `stat_max`, `stat_mean`, `stat_std`
     - Évolution extraite: `evolution_transition`, `evolution_trend`, `evolution_trend_coefficient`
     - Données complètes: `observation_data` (JSON de backup avec toutes les données)
     - Métadonnées: `computed_at` (timestamp)
     - Contrainte UNIQUE: `(exec_id, test_name, params_config_id)` garantit pas de doublons
- **Points critiques**:
  1. **Dual storage**: Données à deux niveaux - colonnes extraites pour requêtes rapides + JSON complet pour flexibilité.
  2. **Conformité Charter 5.5**: Structure alignée avec le standard d'observation.
  3. **Optimisation verdict**: Statistiques et évolution pré-extraites pour analyse rapide.
  4. **Traçabilité complète**: Liens vers `exec_id` et configuration utilisée.
  5. **Gestion d'erreurs**: Statut et message d'erreur inclus.
- **Questions ouvertes**:
  1. Comment sont structurées les données JSON dans `observation_data`?
  2. Comment les statistiques extraites sont-elles calculées à partir des données brutes?
  3. Y a-t-il un mécanisme de versioning pour le schéma?

### ./tests/utilities/discovery.py
- **Rôle**: Découverte automatique et validation des modules de test selon le standard Charter 5.5.
- **Fonctions principales**:
  - `discover_active_tests()`: Parcourt les fichiers test_*.py, les importe, valide leur structure et retourne un dictionnaire des tests actifs.
  - `validate_test_structure(module)`: Valide qu'un module respecte le standard Charter 5.5 (attributs requis, formats, etc.).
- **Dépendances internes**: Aucune (modules internes du projet)
- **Dépendances externes**: importlib, inspect, pathlib, warnings, re (implicite)
- **Points critiques**:
  1. **Strict validation**: La validation est très stricte (9 assertions) et peut rejeter des tests légitimes si mal formatés.
  2. **Performance**: Parcourt le système de fichiers à chaque appel (peut être coûteux si beaucoup de fichiers).
  3. **Gestion d'erreurs**: Utilise `warnings.warn` pour les erreurs d'import/structure, ne bloque pas l'exécution.
  4. **Versioning**: Exige spécifiquement TEST_VERSION == "5.5" (incompatible avec d'autres versions).
  5. **Format TEST_ID**: Format CAT-NNN (3-4 lettres majuscules, tiret, 3 chiffres) très rigide.
- **Structure des tests attendue**:
  - Attributs requis: TEST_ID, TEST_CATEGORY, TEST_VERSION, APPLICABILITY_SPEC, COMPUTATION_SPECS
  - TEST_ID format: "CAT-NNN" (ex: "UNIV-001", "SYM-001")
  - COMPUTATION_SPECS: dict de 1 à 5 métriques, chaque métrique avec 'registry_key' et 'default_params'
  - Pas d'attributs legacy: FORMULAS, is_applicable(), compute_metric()

### ./tests/utilities/applicability.py
- **Rôle**: Évaluation de l'applicabilité des tests à une exécution spécifique basée sur des contraintes définies dans APPLICABILITY_SPEC.
- **Fonctions principales**:
  - `check(test_module, run_metadata)`: Vérifie si un test est applicable aux métadonnées d'une exécution.
  - `add_validator(name, validator)`: Permet d'étendre le système avec des validateurs personnalisés.
- **Dépendances internes**: Aucune (modules internes du projet)
- **Dépendances externes**: typing (Dict, Any, Callable, Tuple)
- **Validators prédéfinis**:
  1. `requires_rank`: Vérifie le rang (dimensions) du state_shape. `None` = pas de contrainte.
  2. `requires_square`: Vérifie si le state_shape est carré (2D et dimensions égales).
  3. `allowed_d_types`: Filtre par type d'encodage D (ex: "ALL" ou liste de types).
  4. `requires_even_dimension`: Vérifie si toutes les dimensions sont paires. `False` = pas de contrainte.
  5. `minimum_dimension`: Vérifie les dimensions minimales. `None` = pas de contrainte.
- **Points critiques**:
  1. **Sémantique des valeurs**: Comportement spécial pour `None` (pas de contrainte) et `False` (désactivé) selon le validateur.
  2. **Extensibilité**: Architecture extensible via `add_validator`.
  3. **Gestion d'erreurs**: Capture les exceptions et retourne des messages d'erreur explicites.
  4. **Parsing d_encoding_id`: Split sur '-' pour extraire le type (ligne 15).
  5. **Validation partielle**: S'arrête à la première contrainte non satisfaite.
- **Questions ouvertes**:
  1. Quels sont les types d'encodage D possibles (d_encoding_id)? (✓ "ALL" est utilisé dans test_univ_002.py)
  2. Comment sont générées les métadonnées `run_metadata` (format complet)? (✓ partiellement répondu par schema_raw.sql)

- **Dépendances internes**:
  - `tests.utilities.registries.registry_manager` (module actif - ✓ analysé)

  2. **Gestion d'erreurs par métrique**: Les erreurs sur une métrique à une itération n'arrêtent pas l'exécution (saut d'itération).

  4. **Chargement config YAML**: Utilisation d'un config_loader pour charger les paramètres par test et catégorie.
  5. **Analyse d'évolution complexe**: Utilisation de plusieurs seuils (explosion, stabilité, croissance) pour catégoriser les transitions.

### ./prc_framework/tests/utilities/registries/post_processors.py
- **Rôle**: Registre central des fonctions de post-traitement pour les métriques calculées. Fournit des transformations standard (arrondi, valeur absolue, log, clipping, notation scientifique) applicables aux résultats numériques.
- **Fonctions principales**:
  - `POST_PROCESSORS` (dict): Dictionnaire global des post-processeurs disponibles, mappant des clés à des fonctions lambda.
  - `get_post_processor(key)`: Récupère une fonction de post-traitement par sa clé. Lève KeyError si inconnue.
  - `add_post_processor(key, func)`: Ajoute un post-processeur personnalisé au registre. Lève ValueError si clé déjà existante.
- **Dépendances internes**: Aucune (modules internes du projet)
- **Dépendances externes**: numpy (pour np.log, np.log10, np.log1p, np.clip), typing
- **Catégories de post-processeurs**:
  1. **Identité**: `identity` (passe-plat)
  2. **Arrondi**: `round_2`, `round_4`, `round_6` (arrondi à N décimales)
  3. **Valeur absolue**: `abs`
  4. **Logarithmique**: `log` (log naturel), `log10` (log base 10), `log1p` (log(1+x) stable)
  5. **Clipping**: `clip_01` (borné entre 0 et 1), `clip_positive` (max(0, x))
  6. **Notation scientifique**: `scientific_3` (format .3e)
- **Points critiques**:
  1. **Stabilité numérique**: Les fonctions logarithmiques ajoutent un epsilon (1e-10) pour éviter log(0).
  2. **Conversion forcée**: Toutes les fonctions convertissent l'entrée en float avant traitement.
  3. **Extensibilité**: API simple pour ajouter des post-processeurs personnalisés.
  4. **Gestion d'erreurs**: Les post-processeurs ne gèrent pas les valeurs aberrantes (NaN, inf) en entrée.
  5. **Format scientifique**: `scientific_3` utilise un formatage string puis reconversion en float, pouvant entraîner une perte de précision.
- **Questions ouvertes**:
  1. Pourquoi `log1p` n'utilise pas d'epsilon comme `log` et `log10`? (Réponse: `log1p` est spécifiquement conçu pour calculer log(1+x) avec précision pour x proche de 0)
  2. Y a-t-il des post-processeurs plus avancés (normalisation, scaling, etc.)?
  3. Comment sont sélectionnés les post-processeurs dans les spécifications de calcul? (✓ démontré dans test_univ_002.py avec 'post_process': 'round_4')

## Questions transversales et risques identifiés
1. **Architecture partiellement désactivée**: Plusieurs composants clés (verdict) sont commentés, ce qui suggère soit un work in progress, soit un mode dégradé.
2. **Base de données critique**: Le programme dépend fortement de deux bases SQLite. Une corruption ou un schéma incompatible pourrait bloquer tout le pipeline. (✓ schémas maintenant documentés)
3. **Dépendances circulaires potentielles**: Les imports de modules `tests.utilities.*` pourraient créer des dépendances complexes.
4. **Performance données**: Chargement de l'historique complet (`load_execution_history`) pourrait être coûteux en mémoire pour de longues exécutions. (✓ schéma raw optimise avec compression et métriques pré-calculées)
5. **Couplage fort avec Charter 5.5**: Le système de discovery impose un format très rigide aux tests, rendant difficile l'évolution ou la rétrocompatibilité.
6. **Validation sémantique inconsistante**: Les validateurs d'applicabilité ont des sémantiques différentes pour les valeurs spéciales (`None`, `False`).

8. **Dépendance à numpy**: Toute l'analyse numérique repose sur numpy, ce qui est robuste mais crée une dépendance forte.
9. **Système de configuration complexe**: La hiérarchie globale/spécifique et la fusion des configurations YAML ajoutent de la complexité.
10. **Architecture de registre complexe**: Le système à trois niveaux (BaseRegistry → Registres concrets → RegistryManager) est puissant mais ajoute de la complexité cognitive.
11. **Post-processing limité**: Les post-processeurs actuels sont principalement des transformations mathématiques simples, manquant potentiellement de transformations statistiques plus avancées.
12. **Documentation inégale**: Certains modules sont bien documentés (algebra_registry), d'autres moins (batch_runner avec TODOs).
13. **Réutilisation de fonctions**: Les tests peuvent appeler plusieurs fois la même fonction avec des paramètres différents (comme démontré dans test_univ_002).
14. **Volume de données**: La table Snapshots avec BLOB compressés peut atteindre des tailles importantes selon le nombre d'itérations et la taille des matrices.
