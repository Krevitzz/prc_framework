## SECTION 0 : axiomes du PRC
### A1: Il existe une dissymétrie informationnelle primitive,
notée D, irréductible à toute cause plus fondamentale accessible.

### A2: Il existe un mécanisme d'interaction locale,
noté Γ, qui agit sur D.

### A3: Aucune transformation globale et stable Γ
ne peut annuler D de manière complète.
```


## SECTION 1 : Hiérarchie des niveaux (INCHANGÉE)

```
\LEVELS{
  L0 = ontological      # Axiomes immutables (A1, A2, A3)
  L1 = epistemic        # Définitions abstraites (D, Γ)
  L2 = theoretical      # Hypothèses testables
  L3 = operative        # Toy models, tests, programmes
  L4 = documentary      # Logs, résultats, analyses
}
```

**Règle stricte** : Un élément L(n) ne peut référencer que L(n-1) ou inférieur.

## SECTION 2 : Architecture du core 

### 2.1 Principe absolu

**Le core PRC = 2 fonctions aveugles (et seulement 2)**

Le core est un moteur d'exécution universel sans aucune logique conditionnelle sur le contenu des données qu'il manipule.

### 2.2 Règles contractuelles du core

#### RÈGLE K1 — Aveuglement total

Le core :
- ne connaît ni la forme de D
- ni sa dimension
- ni son rang
- ni ses symétries
- ni son interprétation

Il manipule des objets opaques (np.ndarray).

#### RÈGLE K2 — Procédure unique et invariante

Le core applique toujours la même procédure :
```
load → apply → record → repeat
```
sans branchement dépendant de D ou Γ.

#### RÈGLE K3 — Zéro validation sémantique

Le core ne valide AUCUNE propriété du contenu :
- pas de symétrie
- pas de normalisation
- pas de bornes
- pas de conservation

Toute validation est dans les TM, jamais dans le core

#### RÈGLE K4 — Neutralité structurelle absolue

Le core accepte tout ce que Γ accepte.

Il n'impose :
- ni forme
- ni invariants
- ni projections

#### RÈGLE K5 — Séparation stricte exécution / interprétation

Le core ne produit pas des résultats, il produit des données.

Les résultats n'apparaissent que dans les TM via des pipelines d'analyse séparés.

Les TM n’imposent aucune propriété à Γ ou D; ils évaluent empiriquement si l’application répétée de Γ conduit à une dégénérescence 
structurelle (trivialité, effondrement de diversité, disparition systématique de symétries pré-existantes) ou à 
l’émergence de régularités non triviales.

### 2.3 Les 2 fonctions du core

#### Fonction 1 : prepare_state

**Signature** :
```python
def prepare_state(base: np.ndarray,
                  modifiers: Optional[List[Callable]] = None) -> np.ndarray
```

**Responsabilité** :
Composer un état D à partir d'une base et de modificateurs.

**Aveuglement** :
- Ne connaît ni la dimension de `base`
- Ne connaît ni le contenu des `modifiers`
- Ne connaît ni leur interprétation

**Opération** :
Application séquentielle aveugle : `state = modifier_n(...modifier_2(modifier_1(base)))`

---

#### Fonction 2 : run_kernel

**Signature** :
```python
def run_kernel(initial_state: np.ndarray,
               gamma: Callable[[np.ndarray], np.ndarray],
               max_iterations: int = 10000,
               convergence_check: Optional[Callable] = None,
               record_history: bool = False) -> Generator
```

**Responsabilité** :
Appliquer itérativement `state_{n+1} = gamma(state_n)`.

**Aveuglement** :
- Ne connaît ni `gamma` (juste l'applique)
- Ne connaît ni `state` (juste le propage)
- Ne connaît ni leur dimension, structure, ou interprétation

**Opération** :
Génération d'états successifs avec contrôle d'arrêt externe (TM).

**Retour** :
```python
if record_history:
    yield (iteration, state, history)
else:
    yield (iteration, state)
```

### 2.4 Ce qui N'est PAS dans le core

❌ Validation de symétrie  
❌ Validation de bornes  
❌ Validation de positivité  
❌ Enforcement de contraintes  
❌ Classes d'état (State, etc.)  
❌ Classes d'opérateurs (PureSaturation, etc.)  
❌ Utilitaires de création (identity, uniform, etc.)  
❌ Sérialisation  
❌ Analyse  
❌ Visualisation  

**Tout cela va dans des modules séparés.**

Section 2.5 "Localisation de la validation"
```
\VALIDATION_RULES{
  core = none                  # Aucune validation
  encodings = creation_only    # Validation à la création, pas après
  modifiers = passthrough      # Aucune validation
  operators = passthrough      # Aucune validation
  tests = observation_only     # Détecte mais ne corrige jamais
}

## SECTION 4 : Nomenclature des états D 

### 4.1 Types de D

Le framework distingue plusieurs types d'états D, tous représentés comme `np.ndarray` mais avec des origines différentes.

#### D^(base)

**Définition** : Champ des possibles maximal pour un test donné.

**Exemples** :
- D^(base) symétrique rang 2 : matrice NxN symétrique
- D^(base) asymétrique rang 2 : matrice NxN asymétrique
- D^(base) rang 3 : tenseur NxNxN

**Localisation** : Défini dans `D_encodings/`

**Usage** :
```python
from encodings.rank2_symmetric import create_identity
D_base = create_identity(n_dof=50)
```

---

#### D^(bruit)

**Définition** : Variations aléatoires appliquées sur D^(base).

**But** : Tester la robustesse/stabilité des patterns émergents.

**Exemples** :
- Bruit Gaussien : `add_gaussian_noise(sigma=0.05)`
- Bruit uniforme : `add_uniform_noise(amplitude=0.01)`

**Localisation** : Défini dans `modifiers/noise.py`

**Usage** :
```python
from modifiers.noise import add_gaussian_noise
D_final = prepare_state(D_base, [add_gaussian_noise(sigma=0.05)])
```

---

#### D^(topo_x)

**Définition** : Contraintes expérimentales restreignant les DOF, patterns pré-identifiés injectés dans D^(base)

**But** : Simuler l'effet de contraintes physiqu.

**Exemples (placeholders)** :
- Double fente : réduction corrélations trans-barrière
- Périodicité : contrainte topologique

**Localisation** : Défini dans `modifiers/constraints.py`

**Note** : Toute la rétro-ingénierie réalité → information reste à faire.

---

#### D^(plugin)

**Définition** : un ou plusieurs pattern identifiés comme étant ascociés à unr observable (lois, objet, etc).

**But** : Tester la persistance de patterns connus.

**Exemples ** :
- Plugin gravitation
- Plugin inertie
- Plugin champ électromagnétique

**Localisation** : `modifiers/plugins/`

**Note** : Nécessite d'abord identification de Γ viables et outils de reconnaissance de patterns.

---

#### D^(domaine)

**Définition** : Compilation pré-remplie de plusieurs plugins.

**Exemples ** :
- D^(mécanique classique) = gravitation + inertie + ...
- D^(électromagnétisme) = charge + champ + ...

**Localisation** : `modifiers/domains/`

---

### 4.2 Composition de D dans un TM

**Principe** : Appel séquentiel de `prepare_state()` avec liste de modifiers.

**Exemple** :
```python
from core.state_preparation import prepare_state
from encodings.rank2_symmetric import create_identity
from modifiers.noise import add_gaussian_noise

D_base = create_identity(50)
D_final = prepare_state(D_base, [
    add_gaussian_noise(sigma=0.05),      # D^(bruit)
])
```

**Note** : Le core (`prepare_state`) ne connaît rien des modifiers, il les applique aveuglément.

## SECTION 5 : Conventions d'identification 

### 5.1 Structure ID

**Format** : `TYPE-SUBTYPE-NNN`

**Types principaux** :
- `HYP` = Hypothesis (L2)
- `TM` = Toy Model (L3)
- `TEST` = Test Protocol (L3) 
- `LOG` = Log Entry (L4)
- `RES` = Result Entry (L4)
- `JOUR` = Journal Entry (L4)
- `CON` = Constraint (L2, dérivée)

**Sous-types standards** :
- `GAM` = relatif à Γ
- `D` = relatif à D
- `PAT` = relatif aux patterns

### 5.2 Valeurs STATUS

```
\STATUS_VALUES{
  WIP = "Work In Progress"
    - WIP[R0-open] : Exploration rang 0 en cours
    - WIP[R0-closed] : Rang 0 terminé, rang ≥1 non testé
    - WIP[Rn-open] : Exploration rang n en cours
  
  SURVIVES = "Non éliminée par les tests actuels"
    - SURVIVES[R0] : Persiste à rang 0
    - SURVIVES[Rn] : Persiste à rang n
  
  REJECTED = "Hypothèse réfutée ou test échoué"
    - REJECTED[R0] : Éliminée comme mécanisme autonome
    - REJECTED[Rn] : Éliminée à rang n
    - REJECTED[GLOBAL] : Éliminée définitivement tous rangs
}
```

*note* une hypothèse de rang 1 peut contenir une plusieurs hypthèse de type WIP[R0-open].

## SECTION 6 : Templates par niveau 

### 6.0 NIVEAU L0 - Axiomes (immutable)

**Contenu** :
- A1: Il existe une dissymétrie informationnelle primitive, notée D, irréductible à toute cause plus fondamentale accessible.
- A2: Il existe un mécanisme d'interaction locale, noté Γ, qui agit sur D.
- A3: Aucune transformation globale et stable Γ ne peut annuler D de manière complète. Cet axiome est plus un filtre épistémique d'un interdit mathématique, les Gamma qui annulent complètement D sont non pertinents expérimentalement.

**Pas de template** : Ces axiomes ne sont jamais modifiés.

### 6.1 NIVEAU L1 - Définitions épistémiques

```
\DEFINITION{
  ID = DEF-[NAME]
  LEVEL = L1
  DEPENDENCIES = {
    axioms = [list]
  }
  STATEMENT = {
    [Description abstraite de l'entité définie]
  }
  NOTES = [optional clarifications]
}
```

**Exemple** :
```
\DEFINITION{
  ID = DEF-GAMMA
  LEVEL = L1
  DEPENDENCIES = {
    axioms = [A2]
  }
  STATEMENT = {
    Γ est un opérateur de compensation locale agissant sur D.
    Γ transforme une configuration informationnelle C_t en C_{t+1}.
  }
  NOTES = {
    - Γ n'optimise rien, n'a aucun but.
    - Γ peut posséder une mémoire finie d'états antérieurs (non-markovien).
    - Γ peut être contraint (localité, efficacité limitée).
  }
}
```

**Interdictions L1** :
- ❌ Aucune forme mathématique concrète (→ L2)
- ❌ Aucun toy model (→ L3)
- ❌ Aucune interprétation physique

  }
  
  THRESHOLD = {
    [paramètres et seuils]
  }

    fields = ["status", "value", "message", "blocking"]
  }
  
  USAGE_EXAMPLE = {
    [code d'exemple]
  }

```

**Exemple** :
```
\TEST_MODULE{
  ID = TEST-SYMMETRY-PRESERVATION
  LEVEL = L3
  CATEGORY = "symmetry"
  
  APPLICABILITY = {
    D_types = ["base_symmetric"]
    gamma_types = ["any"]
    description = "Teste si Γ préserve la symétrie d'un D symétrique"
  }
  
  PROCEDURE = {
    Pour chaque état dans l'historique :
      Calculer asymmetry = ||C - C^T||
      Si asymmetry > TOL : échec
  }
  
  THRESHOLD = {
    TOL_SYMMETRY = 1e-6
  }
  
  OUTPUT = {
    type = "TestResult"
    fields = {
      status = "PASS" | "FAIL"
      value = max_asymmetry
      message = texte descriptif
      blocking = false  # Ce test n'est PAS bloquant par défaut
    }
  }
  
  USAGE_EXAMPLE = {
    from tests.utilities.test_symmetry import test_symmetry_preservation
    result = test_symmetry_preservation(history)
  }
}
```

**IMPORTANT** : Un test observe sans imposer. Si Γ crée asymétrie, c'est une observation intéressante, pas un rejet automatique.

### 7.5 Principe des tests génériques

**RÈGLE FONDAMENTALE** :
Tous les tests sont **observations**, jamais des **prescriptions**.

Un test observe une propriété (symétrie, norme, etc.) et rapporte si elle est :
- **Préservée** : présente initialement, reste présente
- **Créée** : absente initialement, devient présente
- **Détruite** : présente initialement, disparaît
- **Absente** : ni initialement ni finalement

**AUCUN test ne juge** si c'est "bien" ou "mal".  
Le TM décide de l'interprétation.

### 7.6 Tests symétr

**Fichier** : `tests/utilities/test_symmetry.py`
```python
def test_symmetry_status(history: List[np.ndarray]) -> SymmetryReport:
    """
    Observe l'évolution de la symétrie.
    
    Returns:
        SymmetryReport avec:
        - initial_symmetric: bool
        - final_symmetric: bool
        - transition: "preserved" | "created" | "destroyed" | "absent"
        - max_asymmetry: float
    """
    
    def is_symmetric(C, tol=1e-6):
        return np.linalg.norm(C - C.T) < tol
    
    initial = is_symmetric(history[0])
    final = is_symmetric(history[-1])
    
    # Calcule asymétrie maximale
    asymmetries = [np.linalg.norm(C - C.T) for C in history]
    max_asym = max(asymmetries)
    
    # Détermine transition
    if initial and final:
        transition = "preserved"
    elif not initial and final:
        transition = "created"
    elif initial and not final:
        transition = "destroyed"
    else:
        transition = "absent"
    
    return SymmetryReport(
        initial_symmetric=initial,
        final_symmetric=final,
        transition=transition,
        max_asymmetry=max_asym
    )
```

**Usage dans TM** :
```python
# TM décide de l'interprétation
sym_report = test_symmetry_status(history)

if sym_report.transition == "destroyed":
    if D_was_base_symmetric:
        # C'est un échec pour Γ sur D symétrique
        verdict = "FAIL: Γ breaks expected symmetry"
    else:
        # C'est une observation intéressante
        verdict = "PASS: Γ shows expected asymmetry"
```

### 7.7 Tests norme (exemple générique)

**Fichier** : `tests/utilities/test_norm.py`
```python
def test_norm_evolution(history: List[np.ndarray], 
                        norm_type: str = "frobenius") -> NormReport:
    """
    Observe l'évolution de la norme.
    
    Args:
        history: Historique des états
        norm_type: "frobenius" | "spectral" | "max"
    
    Returns:
        NormReport avec:
        - initial_norm: float
        - final_norm: float
        - evolution: "increasing" | "decreasing" | "stable" | "oscillating"
        - max_norm: float
        - min_norm: float
    """
    
    # Calcule normes
    if norm_type == "frobenius":
        norms = [np.linalg.norm(C, 'fro') for C in history]
    elif norm_type == "spectral":
        norms = [np.max(np.abs(np.linalg.eigvalsh(C))) for C in history]
    elif norm_type == "max":
        norms = [np.max(np.abs(C)) for C in history]
    
    # Détecte tendance
    trend = np.polyfit(range(len(norms)), norms, 1)[0]
    
    if abs(trend) < 1e-6:
        evolution = "stable"
    elif trend > 0:
        evolution = "increasing"
    else:
        evolution = "decreasing"
    
    # Détecte oscillations
    diffs = np.diff(norms)
    sign_changes = np.sum(np.diff(np.sign(diffs)) != 0)
    if sign_changes > len(norms) * 0.2:
        evolution = "oscillating"
    
    return NormReport(
        initial_norm=norms[0],
        final_norm=norms[-1],
        evolution=evolution,
        max_norm=max(norms),
        min_norm=min(norms),
        trend=trend
    )
```

**Usage** :
```python
norm_report = test_norm_evolution(history, "frobenius")

if norm_report.evolution == "increasing":
    if norm_report.final_norm > 100:
        verdict = "FAIL: Explosive growth"
    else:
        verdict = "PASS: Controlled growth"
```

## SECTION 8 : Système de rangs et statuts

### 8.1 Définition des rangs

**Rang 0 (R0)** : Test d'un mécanisme isolé (autonome)

Une hypothèse de rang 0 est testée de manière **isolée**, sans composition avec d'autres mécanismes.

**Caractéristiques** :
- Un seul mécanisme élémentaire
- Aucun couplage
- Aucun héritage d'hypothèses antérieures
- Test d'admissibilité minimale

**Exemple** :
```
HYP-GAM-001 : Γ = saturation pure pointwise
→ Testé seul, sans régulation, sans mémoire, sans composition
→ Exploration de rang 0
```

---

**Rang ≥1 (Rn)** : Test de compositions/enrichissements

Une hypothèse de rang n≥1 combine ou enrichit des mécanismes déjà explorés à rang inférieur.

**Caractéristiques** :
- Composition de mécanismes
- Couplages entre opérateurs
- Héritage de contraintes connues
- Test de compatibilité/convergence

**Exemples** :
```
HYP-GAM-002 [R1] : Γ = saturation + renforcement hebbian
→ Combine deux mécanismes élémentaires
→ Exploration de rang 1

HYP-GAM-005 [R2] : Γ = (saturation + hebbian) + mémoire ordre-1
→ Enrichit une composition de rang 1
→ Exploration de rang 2
```

---

### 8.2 Statuts granulaires normalisés

**Format canonique** :
```
STATUS = [ÉTAT][RANG]
```

**États disponibles** :
- `WIP` : Work In Progress
- `SURVIVES` : Non éliminé par tests
- `REJECTED` : Éliminé

**Qualificateurs de rang** :
- `[R0-open]` : Rang 0 en cours d'exploration
- `[R0-closed]` : Rang 0 exploration terminée
- `[Rn]` : Résultat à rang n spécifique
- `[GLOBAL]` : Rejet définitif tous rangs

---

### 8.3 Hiérarchie stricte des rejets
```
REJECTED[R0] < REJECTED[R1] < ... < REJECTED[GLOBAL]
```

**Règle fondamentale** :
Un rejet à rang n est LOCAL à ce test spécifique.
Seul `REJECTED[GLOBAL]` est définitif tous contextes.

**Exemples** :
```
HYP-001: Γ = saturation pure
  REJECTED[R0] → Échoue comme mécanisme isolé
  (Peut être testé dans HYP-010[R1] = saturation + régulation)

HYP-002: Γ = diffusion pure  
  REJECTED[R0] → Échoue comme mécanisme isolé
  (Peut être testé dans HYP-010[R1] = diffusion + saturation)

HYP-003: Γ = croissance explosive incontrôlable
  REJECTED[GLOBAL] → Violation invariants fondamentaux
  (Ne sera JAMAIS retesté, même en composition)
```

**IMPORTANT** :
- `REJECTED[R0]` ne garantit PAS d'être "bon composant"
- C'est à `HYP[R1]` de CHOISIR quels `REJECTED[R0]` combiner
- Aucune obligation de tester toutes les compositions possibles

---

### 8.4 Statuts WIP détaillés

**WIP[R0-open]**
```
Hypothèse en cours d'exploration à rang 0.
Tests partiels, résultats incomplets.
```

**WIP[R0-closed]**
```
HYP testée à rang 0, résultats non concluants.

Signification:
  - Test R0 terminé sans rejet définitif
  - Aucun verdict clair (ni SURVIVES ni REJECTED)
  - Mise en attente indéfinie
  - Peut être reprise dans HYP[R≥1] futures si pertinent

Ce statut n'implique PAS:
  - Viabilité comme composant
  - Obligation de test à rang supérieur
  - Priorité dans exploration future

Critères typiques WIP[R0-closed]:
  - Réussite partielle des tests (certains PASS, certains FAIL non bloquants)
  - Comportement ambigu nécessitant contexte additionnel
  - Ressources/temps insuffisants pour verdict définitif
```

**Nature du statut WIP** :

WIP n'est PAS un label de qualité.

Ce que WIP signifie:
  ✓ Tests en cours ou suspendus
  ✓ Pas encore de verdict définitif
  ✓ Informations insuffisantes pour conclure

Ce que WIP ne signifie PAS:
  ✗ "Prometteur"
  ✗ "Viable si combiné"
  ✗ "Candidat prioritaire pour R≥1"
  ✗ "Partiellement validé"

**Analogie** :
WIP = dossier médical "en attente de diagnostic"
- Pas "malade mais traitable"
- Pas "probablement sain"
- Juste "données insuffisantes, à reprendre ou archiver"

**Transitions typiques depuis WIP** :
```
WIP[R0-open] → REJECTED[R0]
  (Diagnostic clair: inadmissible)

WIP[R0-open] → SURVIVES[R0]
  (Diagnostic clair: admissible)

WIP[R0-open] → WIP[R0-closed]
  (Diagnostic suspendu: ambigu, ressources épuisées)
```

**Destin probable d'un WIP[R0-closed]** :
- 70% cas: Jamais retesté (oublié, non prioritaire)
- 20% cas: Inclus dans HYP[R1] par opportunité
- 10% cas: Réévalué avec nouveaux critères

---

### 8.5 Statuts SURVIVES détaillés

**SURVIVES[R0]**
```
Hypothèse non éliminée à rang 0.
Peut être testée à rang supérieur ou utilisée en composition.

Critères :
- Aucun échec bloquant à rang 0
- Propriétés structurelles satisfaisantes
- Potentiel de contribution identifié
```

**SURVIVES[Rn], n≥1**
```
Hypothèse non éliminée à rang n.
Continue de persister malgré tests de composition.
```

---

### 8.6 Statuts REJECTED détaillés

**REJECTED[R0]**
```
HYP éliminée dans le contexte de test de rang 0.

Signification:
  - Le test spécifique à R0 a échoué
  - Aucune conclusion sur comportement à R≥1
  - Aucune promotion automatique vers rang supérieur
  - Peut être revisitée dans nouvelle HYP[R1] à discrétion

Ce statut n'est PAS:
  - Une recommandation d'utilisation comme composant
  - Une promesse de test futur en composition
  - Une indication de "potentiel caché"
```

**REJECTED[GLOBAL]**
```
Hypothèse éliminée à tous les rangs.

Critère strict :
Viole des invariants non négociables dans TOUT contexte,
y compris composition et enrichissement.

Exemple :
- Croissance explosive incontrôlable
- Destruction systématique de symétrie
- Incompatibilité mathématique fondamentale
```

---

### 8.7 Transitions de statut

**Autorisées** :
```
WIP[R0-open] → WIP[R0-closed]
WIP[R0-open] → SURVIVES[R0]
WIP[R0-open] → REJECTED[R0]

WIP[R0-closed] → WIP[R1-open]    (promotion verticale)
WIP[R0-closed] → REJECTED[GLOBAL] (rejet global post-analyse)

SURVIVES[Rn] → WIP[Rn+1-open]    (promotion verticale)
SURVIVES[Rn] → REJECTED[Rn+1]    (échec au rang supérieur)

REJECTED[Rn] → REJECTED[GLOBAL]  (après épuisement vertical)
```

**Interdites (violation traçabilité)** :
```
✗ WIP[R0-closed] → WIP[R0-open]      (pas de retour arrière)
✗ REJECTED[Rn] → SURVIVES[Rn]        (pas de réactivation)
✗ REJECTED[GLOBAL] → [anything else] (définitif)
```

---

### 8.8 Indépendance des hypothèses de composition

**Principe** :
Chaque hypothèse de rang ≥1 est AUTONOME et SÉLECTIVE.

**Sélectivité des compositions** :
```
HYP-004[R1]: Test de (HYP-001[R0] + HYP-002[R0])
  → Ne dit RIEN sur HYP-003[R0]

HYP-005[R1]: Test de (HYP-003[R0] seul, avec contexte différent)
  → Indépendant de HYP-004

HYP-006[R1]: Test de (HYP-001[R0] + HYP-003[R0])
  → Nouvelle combinaison, résultats imprévisibles
```

**Non-exhaustivité** :

Il n'existe AUCUNE obligation de:
- Tester toutes les paires (HYP-i, HYP-j)
- Tester tous les triplets (HYP-i, HYP-j, HYP-k)
- Épuiser l'espace combinatoire

**Stratégie de sélection** :
- Intuition théorique (mécanismes complémentaires)
- Contraintes émergentes (CON-XXX suggèrent couplages)
- Économie de tests (priorité aux combinaisons prometteuses)

**Traçabilité** :

Chaque HYP[R1] doit documenter explicitement:
- Quelles HYP[R0] sont incluses (champ `parent_hypotheses`)
- Pourquoi cette combinaison spécifique
- Quelles HYP[R0] sont délibérément exclues

**Exemple** :
```
HYP-004[R1] {
  parent_hypotheses = [HYP-001, HYP-002]
  rationale = "Saturation + diffusion = régulation locale"
  excluded = [HYP-003]
  exclusion_reason = "Mécanisme orthogonal, testé séparément"
}
```

### 11.4 Pattern Γ non-markovien

**Γ avec mémoire ordre 1** :

Un Γ non-markovien stocke explicitement l'état précédent.

**Implémentation standard** :
```python
# operators/gamma_hyp_XXX.py

class NonMarkovianGamma:
    """Γ avec mémoire ordre 1 : Γ(C_t, C_t-1)"""
    
    def __init__(self, alpha=0.01):
        self.alpha = alpha
        self._previous_state = None  # Mémoire explicite
    
    def __call__(self, state: np.ndarray) -> np.ndarray:
        """Applique Γ(C_t, C_t-1)"""
        if self._previous_state is None:
            # Première itération : comportement markovien
            result = state.copy()
        else:
            # Itérations suivantes : utilise mémoire
            delta = state - self._previous_state
            result = state + self.alpha * delta
        
        # Stocke état actuel pour prochaine itération
        self._previous_state = state.copy()
        return result
```

**Usage dans TM** :
```python
gamma = NonMarkovianGamma(alpha=0.01)

for i, state in run_kernel(D, gamma, max_iterations=1000):
    # Le kernel appelle gamma(state)
    # Gamma accède automatiquement à son _previous_state interne
    pass
```

**Note importante** :
- Le **core** (`run_kernel`) reste aveugle
- C'est **Γ lui-même** qui gère sa mémoire
- Le kernel n'a pas besoin d'être modifié
