Pipeline global — flux bout en bout
1 batch.py CLI → MODE 1 : run_phase() | MODE 2 : --verdict {phase} (run_verdict_from_parquet) | MODE 3 : --verdict (run_verdict_cross_phases)
2 running.hub_running_new.run_phase() → load_yaml → dry_run_stats → confirmation → registries
3 running.plan_new.generate_kernel_groups() → génère les groupes vmappables (streaming) à partir du YAML et des registries
4 running.plan_new.split_into_batches() → découpe chaque groupe en batches de taille ≤ batch_size
5 running.batching_new.execute_batch() (async) → features_async — construit les inputs batchés et appelle jit(vmap(running.kernel_new._run_fn)) via cache global _vmap_cache
6 running.batching_new.sync_features_b() → np.array() force sync GPU→CPU
7 running.batching_new.rows_from_synced() → construit les rows parquet avec statut (OK/EXPLOSION/NAN_ALL)
8 running.hub_running_new collecte rows_buffer → flush → open_parquet_writer/write_rows_to_parquet → data/results/{phase}.parquet
9 analysing.hub_analysing_v2.run_verdict_from_parquet() → lit parquet via parquet_filter_v2.load_analysing_data(), puis appelle pipeline_v2.run_analysing_pipeline() et écrit les rapports (verdict_v2)
10 analysing.hub_analysing_v2.run_verdict_cross_phases() → compare plusieurs phases via concordance_lite, sortie dans data/results/reports/
Statuts run : OK (features finies), EXPLOSION (Inf), NAN_ALL (toutes NaN non structurelles), INVALID (rank_constraint), FAIL (exception) — définis dans hub_running_new et batching_new.

Workflow run_phase() :
1. load_yaml → extrait phase, dmd_rank, aot_ram_gb
2. dry_run_stats() → affichage + confirmation
3. registries = discover_*_jax()
4. generate_kernel_groups() + rolling_compile_window()
5. Pour chaque kernel_group : split_into_batches()
6. Pour chaque batch : _check_invalid() → si INVALID → make_nan_rows() direct
7. Sinon : execute_batch() async → features_async
8. Pendant que GPU compute, traite le batch précédent (pending) : sync_features_b() + rows_from_synced() → rows_buffer
9. Flush tous les flush_every via open_parquet_writer/write_rows_to_parquet
10. En finally : close_parquet_writer()
Statuts run : OK (features finies), EXPLOSION (Inf), NAN_ALL (toutes NaN), INVALID (rank_constraint), FAIL (exception)
ⓘ aot_ram_gb peut être surchargé par YAML (clé 'aot_ram_gb') — paramètre configurable.
ⓘ dmd_rank lu du YAML (défaut 16) — utilisé dans execute_batch.
ⓘ rolling_compile_window utilise n_workers = max(1, os.cpu_count()-2) — parallélisme CPU/GPU.
▲ double boucle : pending + batch suivant — mais nécessaire pour overlap CPU/GPU.
▲ rows_buffer + flush_every = 1000 — consommation mémoire maîtrisée.
⚠ _check_invalid utilise gamma_registry.get('metadata',{}).get('rank_constraint') — si rank_constraint absent, pas d'invalidation.

running/kernel_new.py
lu ✓
_step_fn(carry, _) → (carry_next, measures) — un pas de dynamique (utilisé dans lax.scan)
_run_fn(gamma_fn, gamma_params, modifier_fn, modifier_params, D_initial, key, max_it, dmd_rank, is_differentiable) → dict features JAX (vmappable)
run_one_jax(...) → dict {str: float} (interface pour exécution unitaire)
Rôle : Noyau compilé unique — applique modificateur, scan gamma, post-traitement en un seul graphe XLA.
_run_fn est la fonction nue vmappable, utilisée par batching_new via jit(vmap(...)).
run_one_jax est une version JIT pour usage unitaire (non batch).
Phases internes :
1. modifier_fn(D_initial, modifier_params, subkey_mod) → D_modified
2. lax.scan sur max_it pas avec carry étendu (state, key, state_prev, sigmas_prev, A_k, P_k) — chaque pas appelle measure_state (depuis jax_features_new)
3. _post_scan_jax(signals, last_state, A_k_final) → dict de features (intégré, signaux jamais matérialisés hors XLA)
Carry étendu :
- state_prev : état t-1 (pour Lyapunov empirique F4, transport F5)
- sigmas_prev : valeurs singulières t-1 (évite un SVD par pas)
- A_k, P_k : carry DMD streaming dans espace spectral (n_dof × n_dof)
ⓘ dmd_rank est un argument statique — toute modification (via YAML) nécessite de vider le cache JAX (jax_cache/).
ⓘ float() sur le premier scalaire dans run_one_jax bloque implicitement jusqu'à la fin du calcul (block_until_ready).
ⓘ La séparation _run_fn / _run_jit permet de vmapper _run_fn sans décorateur jit (batching_new applique jit(vmap(...))).
▲ Le carry contient des matrices (A_k, P_k) de taille n_dof×n_dof — mémoire négligeable par run, mais en batch vmap cela devient (B, n_dof, n_dof). Heureusement, le carry est interne au scan et n'est pas stocké en batch (vmap sur l'axe batch).

analysing/clustering_peeling_v2.py
lu ✓
run_peeling(M_ortho, cfg, mcs_global=None, ms_global=None, M_2d=None, run_regimes=None, output_dir=None, label='peeling', save_debug=False, verbose=False) → Dict
_run_level(...) → (level_result, residual_global)
_homogeneity_score(...) → float
_mcs_residual(...) → int
Rôle : Algorithme de peeling résiduel. Applique HDBSCAN à plusieurs niveaux, avec adaptation des paramètres et score d'homogénéité pour décider si un cluster est définitivement extrait.
Utilisé par clustering_v2.run_clustering.
Principe :
- Niveau 0 : config unique (level_0) sur tous les points.
- Niveaux suivants : batterie de configs (residual_configs) sur le résidu.
- Pour chaque cluster candidat, calcul d'un score d'homogénéité (moyenne des probabilités, silhouette, pureté des régimes).
- Si score ≥ seuil (fonction du niveau), le cluster est extrait (label permanent).
- Sinon, ses points retournent dans le résidu.
- mcs adaptatif pour les niveaux suivants (_mcs_residual).
Détails clés :
- _run_level essaie toutes les configs de la batterie, choisit celle avec le meilleur score composite (nombre de clusters, bruit, ARI bootstrap).
- _homogeneity_score utilise les poids configurés dans cfg['homogeneity'].
- Bootstrap ARI optionnel pour évaluer la stabilité de la config.
- Les résultats de chaque niveau sont tracés (trace) et les labels finaux sont assignés séquentiellement.
ⓘ Le paramètre run_regimes est optionnel ; s'il est fourni, il est utilisé dans le score d'homogénéité (pureté des régimes).
ⓘ Les sorties debug (save_debug=True) sauvegardent labels.npy et un JSON de trace.
ⓘ Le calcul de mcs adaptatif utilise la trace de la covariance comme proxy de la densité.
▲ Le bootstrap ARI peut être coûteux si n_iter est grand ; il est désactivable dans la config.
▲ La boucle sur les configs à chaque niveau peut être lourde si la batterie est grande, mais le nombre de niveaux est limité (max_levels).
analysing/clustering_v2.py
lu ✓
prepare_matrix(data, corr_threshold=0.85, protected_features=None) → (M_ortho, kept_names, matrix_meta)
compute_projection(M_ortho, cfg=None, cache_path=None) → M_2d
run_clustering(M_ortho, feat_names, peeling_cfg, M_2d=None, output_dir=None, label='clustering', save_debug=False, verbose=False) → Optional[Dict]
Rôle : Préparation de la matrice (normalisation, log‑transform, robust scaling, orthogonalisation) et interface vers le peeling.
prepare_matrix :
- Utilise data.features_for_ml() pour exclure les colonnes health_*.
- nan → 0.0, log‑transform si dynamique > 1e6, RobustScaler, puis sélection orthogonale (corr < 0.85, protège PROTECTED_FEATURES_V7).
- Retourne M_ortho (float64) et les noms conservés.
compute_projection :
- PCA(50) puis t‑SNE (perplexity auto = min(50, max(5, sqrt(n)//3)) ou UMAP si n ≥ umap_threshold (défaut 50k) et umap installé.
- Cache .npy optionnel.
run_clustering :
- Appelle clustering_peeling_v2.run_peeling avec les paramètres de la config.
- Calcule mcs = max(peeling_cfg['mcs_floor'], n//30) et ms = max(peeling_cfg['ms_floor'], n//100).
- Retourne un dict contenant labels, n_clusters, n_noise, etc.
ⓘ PROTECTED_FEATURES_V7 est un set de features qui ne sont jamais éliminées lors de l'orthogonalisation.
ⓘ En cas d'échec de RobustScaler (par ex. colonne constante), on utilise les données brutes après log‑transform.
▲ Le calcul de corrélation (O(F² n)) peut être coûteux si F est grand, mais le nombre de features est modéré (~68).

Rôle : Nommage compositionnel des clusters basé sur les distributions des features.

Architecture :
- Les fonctions build_cluster_profile et build_layer_distribution agrègent les statistiques du cluster (médiane, IQR, etc.).
- La classe ClusterNamer lit la config, puis pour chaque slot (ex: AMPLITUDE, VITESSE) applique le handler correspondant (_eval_zones, _eval_delta, etc.) qui retourne un terme et une confiance.
- Les termes avec confiance ≥ conf_min_name deviennent le nom primaire, ceux ≥ conf_min_display sont secondaires.
- Les slots non calibrés (calibrated: false) sont traités par _eval_uncalibrated et apparaissent comme [slot=?].

- zones : zones exclusives avec bornes lo/hi (ex: AMPLITUDE, DMD).
- delta : bidirectionnel autour de 0, avec possibilité d'omettre si neutre (ENT, RNK, LYA).
- threshold : seuil unique, omit_if_clean (CND).
- uncalibrated : placeholder pour slots non calibrés (SEQ).
✚ Ajouter un nouveau slot : une entrée dans le YAML suffit ; le code dispatche via le mode.
✚ Ajouter un nouveau mode : écrire un handler et l'ajouter dans _EVAL_DISPATCH.
ⓘ Le mode uncalibrated est une dette technique documentée : le slot SEQ nécessite des features phasiques (F8) non encore disponibles.
ⓘ La fonction _percentile_rank utilise np.searchsorted pour un calcul vectorisé efficace.
▲ La construction du profil (build_cluster_profile) itère sur toutes les features (F~68) ; reste léger.
