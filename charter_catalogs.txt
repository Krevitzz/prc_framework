1. Nature du catalogue (ce que ce n’est pas / ce que c’est)
❌ Ce que ce catalogue n’est pas

❌ Ce n’est pas une spécification normative
→ Rien dans le catalogue n’est “figé” au sens architectural.

❌ Ce n’est pas une liste exhaustive définitive
→ Les fonctions, modules et signatures sont extensibles.

❌ Ce n’est pas une source de vérité méthodologique
→ Il ne définit ni la logique PRC, ni les invariants cognitifs.

✅ Ce que ce catalogue est

👉 Une cartographie fonctionnelle dynamique de l’écosystème utilitaire PRC, à un instant donné.

On peut le formuler ainsi :

Le catalogue décrit où se trouve chaque responsabilité technique,
mais pas pourquoi ni comment elle doit évoluer.

Le pourquoi et le comment relèvent exclusivement du charter.

2. Rôle cognitif du catalogue dans le système PRC
Le catalogue est une table d’orientation, pas une boussole.

Pour Claude (ou tout agent), le rôle attendu est le suivant :

Si je dois coder quelque chose de nouveau, le catalogue me dit :

quel type de problème existe déjà,

dans quel module ce problème est normalement traité,

quelles structures d’I/O sont attendues,

quelles dépendances sont déjà stabilisées.

Mais :

Le catalogue ne m’autorise jamais à inventer une nouvelle taxonomie,
ni à modifier implicitement les flux ou les contrats.

3. Relation stricte Charter ↔ Catalogue (point clé à expliciter)
Hiérarchie non négociable
CHARTE (figée, cognitive, méthodologique)
        ↓
CATALOGUES (évolutifs, exhaustifs, descriptifs)
        ↓
MODULES / FICHIERS PY (implémentation concrète)


Formulation explicite à donner à Claude :

Le charter définit les invariants (axes, rôles, frontières épistémiques).

Les catalogues listent les implémentations existantes conformes à ces invariants.

Les modules peuvent changer, tant qu’ils restent compatibles avec les deux niveaux supérieurs.

4. Comment Claude doit utiliser le catalogue (règles opérationnelles)
Règle 1 — Jamais de création “hors catalogue”

Si Claude doit :

ajouter un test,

ajouter un axe (modifier, encoding, etc.),

enrichir un profiling,

créer un nouveau writer ou utilitaire,

👉 il doit d’abord identifier dans le catalogue :

le module analogue existant,

la fonction équivalente,

la structure d’entrée/sortie attendue.

Règle 2 — Si information absente → STOP explicite

Point fondamental que tu évoques et qu’il faut écrire noir sur blanc :

Si une information nécessaire n’est pas trouvée dans le charter OU le catalogue,
Claude doit impérativement demander le fichier source concerné.

Exemples à inclure tels quels :

“Le catalogue mentionne aggregation_utils, mais je dois vérifier le calcul exact → merci de fournir le fichier.”

“Le charter impose une séparation calcul/interprétation, mais la fonction concrète n’est pas listée → besoin du module.”

👉 Interdiction d’inférer silencieusement.

5. Lecture correcte du catalogue par type de tâche
A. Créer un nouveau module utilitaire

Claude doit :

Identifier la responsabilité (statistique, I/O, classification, formatage, etc.)

Vérifier si un utilitaire analogue existe déjà.

Respecter :

signatures existantes,

style de retour (dicts normalisés),

absence d’interprétation métier.

👉 Le catalogue sert de gabarit de conception, pas de simple index.

B. Étendre un axe (gamma, modifier, encoding, test)

Le catalogue montre clairement que :

gamma_profiling est un cas particulier d’un schéma plus général,

les utils sont déjà agnostiques de l’axe.

Donc Claude doit comprendre :

“Si je sais profiler gamma, je sais profiler modifier ou encoding,
à condition de réutiliser les mêmes utils, structures et étapes.”

👉 Le catalogue encode la généricité implicite de l’architecture.

C. Lire / comparer des résultats multi-axes

Le catalogue n’impose pas de lecture finale unique, mais il garantit que :

toutes les analyses passent par les mêmes projections,

les mêmes métriques statistiques,

les mêmes régimes et taxonomies.

👉 Cela rend possible le cross-profiling ultérieur, sans refactor.

6. Pourquoi le catalogue seul n’est PAS suffisant (à dire explicitement)

Phrase clé à écrire telle quelle dans le charter 5.6 :

Le catalogue, pris isolément, n’est pas suffisant pour coder correctement.
Il doit toujours être lu à la lumière du charter, qui fixe :

la séparation calcul / interprétation,

la non-causalité des analyses R0,

la stabilité des axes conceptuels,

l’interdiction des glissements taxonomiques.

7. Reformulation synthétique “à destination de Claude”

Si je devais le résumer en un bloc quasi contractuel :

Le catalogue PRC est une cartographie évolutive des modules et fonctions existants.
Il décrit où se trouvent les responsabilités techniques,
mais ne définit ni la méthodologie, ni les invariants conceptuels.

Toute implémentation nouvelle doit :

être compatible avec le charter,

s’inscrire dans un pattern existant du catalogue,

ou, en cas d’ambiguïté, déclencher explicitement une demande de fichier source.

Toute inférence implicite ou création hors de ce cadre est interdite.
