## SECTION 0 : axiomes du PRC
### A1: Il existe une dissymétrie informationnelle primitive,
notée D, irréductible à toute cause plus fondamentale accessible.

### A2: Il existe un mécanisme d'interaction locale,
noté Γ, qui agit sur D.

### A3: Aucune transformation globale et stable Γ
ne peut annuler D de manière complète.
```


## SECTION 1 : Hiérarchie des niveaux (INCHANGÉE)

```
\LEVELS{
  L0 = ontological      # Axiomes immutables (A1, A2, A3)
  L1 = epistemic        # Définitions abstraites (D, Γ)
  L2 = theoretical      # Hypothèses testables
  L3 = operative        # Toy models, tests, programmes
  L4 = documentary      # Logs, résultats, analyses
}
```

**Règle stricte** : Un élément L(n) ne peut référencer que L(n-1) ou inférieur.

## SECTION 2 : Architecture du core 

### 2.1 Principe absolu

**Le core PRC = 2 fonctions aveugles (et seulement 2)**

Le core est un moteur d'exécution universel sans aucune logique conditionnelle sur le contenu des données qu'il manipule.

### 2.2 Règles contractuelles du core

#### RÈGLE K1 — Aveuglement total

Le core :
- ne connaît ni la forme de D
- ni sa dimension
- ni son rang
- ni ses symétries
- ni son interprétation

Il manipule des objets opaques (np.ndarray).

#### RÈGLE K2 — Procédure unique et invariante

Le core applique toujours la même procédure :
```
load → apply → record → repeat
```
sans branchement dépendant de D ou Γ.

#### RÈGLE K3 — Zéro validation sémantique

Le core ne valide AUCUNE propriété du contenu :
- pas de symétrie
- pas de normalisation
- pas de bornes
- pas de conservation

Toute validation est dans les TM, jamais dans le core

#### RÈGLE K4 — Neutralité structurelle absolue

Le core accepte tout ce que Γ accepte.

Il n'impose :
- ni forme
- ni invariants
- ni projections

#### RÈGLE K5 — Séparation stricte exécution / interprétation

Le core ne produit pas des résultats, il produit des données.

Les résultats n'apparaissent que dans les TM via des pipelines d'analyse séparés.

Les TM n’imposent aucune propriété à Γ ou D; ils évaluent empiriquement si l’application répétée de Γ conduit à une dégénérescence 
structurelle (trivialité, effondrement de diversité, disparition systématique de symétries pré-existantes) ou à 
l’émergence de régularités non triviales.

### 2.3 Les 2 fonctions du core

#### Fonction 1 : prepare_state

**Signature** :
```python
def prepare_state(base: np.ndarray,
                  modifiers: Optional[List[Callable]] = None) -> np.ndarray
```

**Responsabilité** :
Composer un état D à partir d'une base et de modificateurs.

**Aveuglement** :
- Ne connaît ni la dimension de `base`
- Ne connaît ni le contenu des `modifiers`
- Ne connaît ni leur interprétation

**Opération** :
Application séquentielle aveugle : `state = modifier_n(...modifier_2(modifier_1(base)))`

---

#### Fonction 2 : run_kernel

**Signature** :
```python
def run_kernel(initial_state: np.ndarray,
               gamma: Callable[[np.ndarray], np.ndarray],
               max_iterations: int = 10000,
               convergence_check: Optional[Callable] = None,
               record_history: bool = False) -> Generator
```

**Responsabilité** :
Appliquer itérativement `state_{n+1} = gamma(state_n)`.

**Aveuglement** :
- Ne connaît ni `gamma` (juste l'applique)
- Ne connaît ni `state` (juste le propage)
- Ne connaît ni leur dimension, structure, ou interprétation

**Opération** :
Génération d'états successifs avec contrôle d'arrêt externe (TM).

**Retour** :
```python
if record_history:
    yield (iteration, state, history)
else:
    yield (iteration, state)
```

### 2.4 Ce qui N'est PAS dans le core

❌ Validation de symétrie  
❌ Validation de bornes  
❌ Validation de positivité  
❌ Enforcement de contraintes  
❌ Classes d'état (State, etc.)  
❌ Classes d'opérateurs (PureSaturation, etc.)  
❌ Utilitaires de création (identity, uniform, etc.)  
❌ Sérialisation  
❌ Analyse  
❌ Visualisation  

**Tout cela va dans des modules séparés.**

Section 2.5 "Localisation de la validation"
```
\VALIDATION_RULES{
  core = none                  # Aucune validation
  encodings = creation_only    # Validation à la création, pas après
  modifiers = passthrough      # Aucune validation
  operators = passthrough      # Aucune validation
  tests = observation_only     # Détecte mais ne corrige jamais
}

## SECTION 4 : Nomenclature des états D 

### 4.1 Types de D

Le framework distingue plusieurs types d'états D, tous représentés comme `np.ndarray` mais avec des origines différentes.

#### D^(base)

**Définition** : Champ des possibles maximal pour un test donné.

**Exemples** :
- D^(base) symétrique rang 2 : matrice NxN symétrique
- D^(base) asymétrique rang 2 : matrice NxN asymétrique
- D^(base) rang 3 : tenseur NxNxN

**Localisation** : Défini dans `D_encodings/`

**Usage** :
```python
from encodings.rank2_symmetric import create_identity
D_base = create_identity(n_dof=50)
```

---

#### D^(bruit)

**Définition** : Variations aléatoires appliquées sur D^(base).

**But** : Tester la robustesse/stabilité des patterns émergents.

**Exemples** :
- Bruit Gaussien : `add_gaussian_noise(sigma=0.05)`
- Bruit uniforme : `add_uniform_noise(amplitude=0.01)`

**Localisation** : Défini dans `modifiers/noise.py`

**Usage** :
```python
from modifiers.noise import add_gaussian_noise
D_final = prepare_state(D_base, [add_gaussian_noise(sigma=0.05)])
```

---

#### D^(topo_x)

**Définition** : Contraintes expérimentales restreignant les DOF, patterns pré-identifiés injectés dans D^(base)

**But** : Simuler l'effet de contraintes physiqu.

**Exemples (placeholders)** :
- Double fente : réduction corrélations trans-barrière
- Périodicité : contrainte topologique

**Localisation** : Défini dans `modifiers/constraints.py`

**Note** : Toute la rétro-ingénierie réalité → information reste à faire.

---

#### D^(plugin)

**Définition** : un ou plusieurs pattern identifiés comme étant ascociés à unr observable (lois, objet, etc).

**But** : Tester la persistance de patterns connus.

**Exemples ** :
- Plugin gravitation
- Plugin inertie
- Plugin champ électromagnétique

**Localisation** : `modifiers/plugins/`

**Note** : Nécessite d'abord identification de Γ viables et outils de reconnaissance de patterns.

---

#### D^(domaine)

**Définition** : Compilation pré-remplie de plusieurs plugins.

**Exemples ** :
- D^(mécanique classique) = gravitation + inertie + ...
- D^(électromagnétisme) = charge + champ + ...

**Localisation** : `modifiers/domains/`

---

### 4.2 Composition de D dans un TM

**Principe** : Appel séquentiel de `prepare_state()` avec liste de modifiers.

**Exemple** :
```python
from core.state_preparation import prepare_state
from encodings.rank2_symmetric import create_identity
from modifiers.noise import add_gaussian_noise

D_base = create_identity(50)
D_final = prepare_state(D_base, [
    add_gaussian_noise(sigma=0.05),      # D^(bruit)
])
```

**Note** : Le core (`prepare_state`) ne connaît rien des modifiers, il les applique aveuglément.

## SECTION 5 : Conventions d'identification 

### 5.1 Structure ID

**Format** : `TYPE-SUBTYPE-NNN`

**Types principaux** :
- `HYP` = Hypothesis (L2)
- `TM` = Toy Model (L3)
- `TEST` = Test Protocol (L3) 
- `LOG` = Log Entry (L4)
- `RES` = Result Entry (L4)
- `JOUR` = Journal Entry (L4)
- `CON` = Constraint (L2, dérivée)

**Sous-types standards** :
- `GAM` = relatif à Γ
- `D` = relatif à D
- `PAT` = relatif aux patterns

### 5.2 Valeurs STATUS

```
\STATUS_VALUES{
  WIP = "Work In Progress"
    - WIP[R0-open] : Exploration rang 0 en cours
    - WIP[R0-closed] : Rang 0 terminé, rang ≥1 non testé
    - WIP[Rn-open] : Exploration rang n en cours
  
  SURVIVES = "Non éliminée par les tests actuels"
    - SURVIVES[R0] : Persiste à rang 0
    - SURVIVES[Rn] : Persiste à rang n
  
  REJECTED = "Hypothèse réfutée ou test échoué"
    - REJECTED[R0] : Éliminée comme mécanisme autonome
    - REJECTED[Rn] : Éliminée à rang n
    - REJECTED[GLOBAL] : Éliminée définitivement tous rangs
}
```

*note* une hypothèse de rang 1 peut contenir une plusieurs hypthèse de type WIP[R0-open].

## SECTION 6 : Templates par niveau 

### 6.0 NIVEAU L0 - Axiomes (immutable)

**Contenu** :
- A1: Il existe une dissymétrie informationnelle primitive, notée D, irréductible à toute cause plus fondamentale accessible.
- A2: Il existe un mécanisme d'interaction locale, noté Γ, qui agit sur D.
- A3: Aucune transformation globale et stable Γ ne peut annuler D de manière complète. Cet axiome est plus un filtre épistémique d'un interdit mathématique, les Gamma qui annulent complètement D sont non pertinents expérimentalement.

**Pas de template** : Ces axiomes ne sont jamais modifiés.

### 6.1 NIVEAU L1 - Définitions épistémiques

```
\DEFINITION{
  ID = DEF-[NAME]
  LEVEL = L1
  DEPENDENCIES = {
    axioms = [list]
  }
  STATEMENT = {
    [Description abstraite de l'entité définie]
  }
  NOTES = [optional clarifications]
}
```

**Exemple** :
```
\DEFINITION{
  ID = DEF-GAMMA
  LEVEL = L1
  DEPENDENCIES = {
    axioms = [A2]
  }
  STATEMENT = {
    Γ est un opérateur de compensation locale agissant sur D.
    Γ transforme une configuration informationnelle C_t en C_{t+1}.
  }
  NOTES = {
    - Γ n'optimise rien, n'a aucun but.
    - Γ peut posséder une mémoire finie d'états antérieurs (non-markovien).
    - Γ peut être contraint (localité, efficacité limitée).
  }
}
```

**Interdictions L1** :
- ❌ Aucune forme mathématique concrète (→ L2)
- ❌ Aucun toy model (→ L3)
- ❌ Aucune interprétation physique

### 6.4 NIVEAU L3 - Test Protocols (MODULAIRES)

```
\TEST_MODULE{

    description = [quand utiliser ce test]
  }
  
  PROCEDURE = {
    [Description du protocole]
  }
  
  THRESHOLD = {
    [paramètres et seuils]
  }
  
  OUTPUT = {
    type = "TestResult"
    fields = ["status", "value", "message", "blocking"]
  }
  
  USAGE_EXAMPLE = {
    [code d'exemple]
  }
}
```

**Exemple** :
```
\TEST_MODULE{
  ID = TEST-SYMMETRY-PRESERVATION
  LEVEL = L3
  CATEGORY = "symmetry"
  
  APPLICABILITY = {
    D_types = ["base_symmetric"]
    gamma_types = ["any"]
    description = "Teste si Γ préserve la symétrie d'un D symétrique"
  }
  
  PROCEDURE = {
    Pour chaque état dans l'historique :
      Calculer asymmetry = ||C - C^T||
      Si asymmetry > TOL : échec
  }
  
  THRESHOLD = {
    TOL_SYMMETRY = 1e-6
  }
  
  OUTPUT = {
    type = "TestResult"
    fields = {
      status = "PASS" | "FAIL"
      value = max_asymmetry
      message = texte descriptif
      blocking = false  # Ce test n'est PAS bloquant par défaut
    }
  }
  
  USAGE_EXAMPLE = {
    from tests.utilities.test_symmetry import test_symmetry_preservation
    result = test_symmetry_preservation(history)
  }
}
```

**IMPORTANT** : Un test observe sans imposer. Si Γ crée asymétrie, c'est une observation intéressante, pas un rejet automatique.

### 7.5 Principe des tests génériques

**RÈGLE FONDAMENTALE** :
Tous les tests sont **observations**, jamais des **prescriptions**.

Un test observe une propriété (symétrie, norme, etc.) et rapporte si elle est :
- **Préservée** : présente initialement, reste présente
- **Créée** : absente initialement, devient présente
- **Détruite** : présente initialement, disparaît
- **Absente** : ni initialement ni finalement

**AUCUN test ne juge** si c'est "bien" ou "mal".  
Le TM décide de l'interprétation.

### 7.6 Tests symétr

**Fichier** : `tests/utilities/test_symmetry.py`
```python
def test_symmetry_status(history: List[np.ndarray]) -> SymmetryReport:
    """
    Observe l'évolution de la symétrie.
    
    Returns:
        SymmetryReport avec:
        - initial_symmetric: bool
        - final_symmetric: bool
        - transition: "preserved" | "created" | "destroyed" | "absent"
        - max_asymmetry: float
    """
    
    def is_symmetric(C, tol=1e-6):
        return np.linalg.norm(C - C.T) < tol
    
    initial = is_symmetric(history[0])
    final = is_symmetric(history[-1])
    
    # Calcule asymétrie maximale
    asymmetries = [np.linalg.norm(C - C.T) for C in history]
    max_asym = max(asymmetries)
    
    # Détermine transition
    if initial and final:
        transition = "preserved"
    elif not initial and final:
        transition = "created"
    elif initial and not final:
        transition = "destroyed"
    else:
        transition = "absent"
    
    return SymmetryReport(
        initial_symmetric=initial,
        final_symmetric=final,
        transition=transition,
        max_asymmetry=max_asym
    )
```

**Usage dans TM** :
```python
# TM décide de l'interprétation
sym_report = test_symmetry_status(history)

if sym_report.transition == "destroyed":
    if D_was_base_symmetric:
        # C'est un échec pour Γ sur D symétrique
        verdict = "FAIL: Γ breaks expected symmetry"
    else:
        # C'est une observation intéressante
        verdict = "PASS: Γ shows expected asymmetry"
```

### 7.7 Tests norme (exemple générique)

**Fichier** : `tests/utilities/test_norm.py`
```python
def test_norm_evolution(history: List[np.ndarray], 
                        norm_type: str = "frobenius") -> NormReport:
    """
    Observe l'évolution de la norme.
    
    Args:
        history: Historique des états
        norm_type: "frobenius" | "spectral" | "max"
    
    Returns:
        NormReport avec:
        - initial_norm: float
        - final_norm: float
        - evolution: "increasing" | "decreasing" | "stable" | "oscillating"
        - max_norm: float
        - min_norm: float
    """
    
    # Calcule normes
    if norm_type == "frobenius":
        norms = [np.linalg.norm(C, 'fro') for C in history]
    elif norm_type == "spectral":
        norms = [np.max(np.abs(np.linalg.eigvalsh(C))) for C in history]
    elif norm_type == "max":
        norms = [np.max(np.abs(C)) for C in history]
    
    # Détecte tendance
    trend = np.polyfit(range(len(norms)), norms, 1)[0]
    
    if abs(trend) < 1e-6:
        evolution = "stable"
    elif trend > 0:
        evolution = "increasing"
    else:
        evolution = "decreasing"
    
    # Détecte oscillations
    diffs = np.diff(norms)
    sign_changes = np.sum(np.diff(np.sign(diffs)) != 0)
    if sign_changes > len(norms) * 0.2:
        evolution = "oscillating"
    
    return NormReport(
        initial_norm=norms[0],
        final_norm=norms[-1],
        evolution=evolution,
        max_norm=max(norms),
        min_norm=min(norms),
        trend=trend
    )
```

**Usage** :
```python
norm_report = test_norm_evolution(history, "frobenius")

if norm_report.evolution == "increasing":
    if norm_report.final_norm > 100:
        verdict = "FAIL: Explosive growth"
    else:
        verdict = "PASS: Controlled growth"
```

## SECTION 8 : Système de rangs et statuts

### 8.1 Définition des rangs

**Rang 0 (R0)** : Test d'un mécanisme isolé (autonome)

Une hypothèse de rang 0 est testée de manière **isolée**, sans composition avec d'autres mécanismes.

**Caractéristiques** :
- Un seul mécanisme élémentaire
- Aucun couplage
- Aucun héritage d'hypothèses antérieures
- Test d'admissibilité minimale

**Exemple** :
```
HYP-GAM-001 : Γ = saturation pure pointwise
→ Testé seul, sans régulation, sans mémoire, sans composition
→ Exploration de rang 0
```

---

**Rang ≥1 (Rn)** : Test de compositions/enrichissements

Une hypothèse de rang n≥1 combine ou enrichit des mécanismes déjà explorés à rang inférieur.

**Caractéristiques** :
- Composition de mécanismes
- Couplages entre opérateurs
- Héritage de contraintes connues
- Test de compatibilité/convergence

**Exemples** :
```
HYP-GAM-002 [R1] : Γ = saturation + renforcement hebbian
→ Combine deux mécanismes élémentaires
→ Exploration de rang 1

HYP-GAM-005 [R2] : Γ = (saturation + hebbian) + mémoire ordre-1
→ Enrichit une composition de rang 1
→ Exploration de rang 2
```

---

### 8.2 Statuts granulaires normalisés

**Format canonique** :
```
STATUS = [ÉTAT][RANG]
```

**États disponibles** :
- `WIP` : Work In Progress
- `SURVIVES` : Non éliminé par tests
- `REJECTED` : Éliminé

**Qualificateurs de rang** :
- `[R0-open]` : Rang 0 en cours d'exploration
- `[R0-closed]` : Rang 0 exploration terminée
- `[Rn]` : Résultat à rang n spécifique
- `[GLOBAL]` : Rejet définitif tous rangs

---

### 8.3 Hiérarchie stricte des rejets
```
REJECTED[R0] < REJECTED[R1] < ... < REJECTED[GLOBAL]
```

**Règle fondamentale** :
Un rejet à rang n est LOCAL à ce test spécifique.
Seul `REJECTED[GLOBAL]` est définitif tous contextes.

**Exemples** :
```
HYP-001: Γ = saturation pure
  REJECTED[R0] → Échoue comme mécanisme isolé
  (Peut être testé dans HYP-010[R1] = saturation + régulation)

HYP-002: Γ = diffusion pure  
  REJECTED[R0] → Échoue comme mécanisme isolé
  (Peut être testé dans HYP-010[R1] = diffusion + saturation)

HYP-003: Γ = croissance explosive incontrôlable
  REJECTED[GLOBAL] → Violation invariants fondamentaux
  (Ne sera JAMAIS retesté, même en composition)
```

**IMPORTANT** :
- `REJECTED[R0]` ne garantit PAS d'être "bon composant"
- C'est à `HYP[R1]` de CHOISIR quels `REJECTED[R0]` combiner
- Aucune obligation de tester toutes les compositions possibles

---

### 8.4 Statuts WIP détaillés

**WIP[R0-open]**
```
Hypothèse en cours d'exploration à rang 0.
Tests partiels, résultats incomplets.
```

**WIP[R0-closed]**
```
HYP testée à rang 0, résultats non concluants.

Signification:
  - Test R0 terminé sans rejet définitif
  - Aucun verdict clair (ni SURVIVES ni REJECTED)
  - Mise en attente indéfinie
  - Peut être reprise dans HYP[R≥1] futures si pertinent

Ce statut n'implique PAS:
  - Viabilité comme composant
  - Obligation de test à rang supérieur
  - Priorité dans exploration future

Critères typiques WIP[R0-closed]:
  - Réussite partielle des tests (certains PASS, certains FAIL non bloquants)
  - Comportement ambigu nécessitant contexte additionnel
  - Ressources/temps insuffisants pour verdict définitif
```

**Nature du statut WIP** :

WIP n'est PAS un label de qualité.

Ce que WIP signifie:
  ✓ Tests en cours ou suspendus
  ✓ Pas encore de verdict définitif
  ✓ Informations insuffisantes pour conclure

Ce que WIP ne signifie PAS:
  ✗ "Prometteur"
  ✗ "Viable si combiné"
  ✗ "Candidat prioritaire pour R≥1"
  ✗ "Partiellement validé"

**Analogie** :
WIP = dossier médical "en attente de diagnostic"
- Pas "malade mais traitable"
- Pas "probablement sain"
- Juste "données insuffisantes, à reprendre ou archiver"

**Transitions typiques depuis WIP** :
```
WIP[R0-open] → REJECTED[R0]
  (Diagnostic clair: inadmissible)

WIP[R0-open] → SURVIVES[R0]
  (Diagnostic clair: admissible)

WIP[R0-open] → WIP[R0-closed]
  (Diagnostic suspendu: ambigu, ressources épuisées)
```

**Destin probable d'un WIP[R0-closed]** :
- 70% cas: Jamais retesté (oublié, non prioritaire)
- 20% cas: Inclus dans HYP[R1] par opportunité
- 10% cas: Réévalué avec nouveaux critères

---

### 8.5 Statuts SURVIVES détaillés

**SURVIVES[R0]**
```
Hypothèse non éliminée à rang 0.
Peut être testée à rang supérieur ou utilisée en composition.

Critères :
- Aucun échec bloquant à rang 0
- Propriétés structurelles satisfaisantes
- Potentiel de contribution identifié
```

**SURVIVES[Rn], n≥1**
```
Hypothèse non éliminée à rang n.
Continue de persister malgré tests de composition.
```

---

### 8.6 Statuts REJECTED détaillés

**REJECTED[R0]**
```
HYP éliminée dans le contexte de test de rang 0.

Signification:
  - Le test spécifique à R0 a échoué
  - Aucune conclusion sur comportement à R≥1
  - Aucune promotion automatique vers rang supérieur
  - Peut être revisitée dans nouvelle HYP[R1] à discrétion

Ce statut n'est PAS:
  - Une recommandation d'utilisation comme composant
  - Une promesse de test futur en composition
  - Une indication de "potentiel caché"
```

**REJECTED[GLOBAL]**
```
Hypothèse éliminée à tous les rangs.

Critère strict :
Viole des invariants non négociables dans TOUT contexte,
y compris composition et enrichissement.

Exemple :
- Croissance explosive incontrôlable
- Destruction systématique de symétrie
- Incompatibilité mathématique fondamentale
```

---

### 8.7 Transitions de statut

**Autorisées** :
```
WIP[R0-open] → WIP[R0-closed]
WIP[R0-open] → SURVIVES[R0]
WIP[R0-open] → REJECTED[R0]

WIP[R0-closed] → WIP[R1-open]    (promotion verticale)
WIP[R0-closed] → REJECTED[GLOBAL] (rejet global post-analyse)

SURVIVES[Rn] → WIP[Rn+1-open]    (promotion verticale)
SURVIVES[Rn] → REJECTED[Rn+1]    (échec au rang supérieur)

REJECTED[Rn] → REJECTED[GLOBAL]  (après épuisement vertical)
```

**Interdites (violation traçabilité)** :
```
✗ WIP[R0-closed] → WIP[R0-open]      (pas de retour arrière)
✗ REJECTED[Rn] → SURVIVES[Rn]        (pas de réactivation)
✗ REJECTED[GLOBAL] → [anything else] (définitif)
```

---

### 8.8 Indépendance des hypothèses de composition

**Principe** :
Chaque hypothèse de rang ≥1 est AUTONOME et SÉLECTIVE.

**Sélectivité des compositions** :
```
HYP-004[R1]: Test de (HYP-001[R0] + HYP-002[R0])
  → Ne dit RIEN sur HYP-003[R0]

HYP-005[R1]: Test de (HYP-003[R0] seul, avec contexte différent)
  → Indépendant de HYP-004

HYP-006[R1]: Test de (HYP-001[R0] + HYP-003[R0])
  → Nouvelle combinaison, résultats imprévisibles
```

**Non-exhaustivité** :

Il n'existe AUCUNE obligation de:
- Tester toutes les paires (HYP-i, HYP-j)
- Tester tous les triplets (HYP-i, HYP-j, HYP-k)
- Épuiser l'espace combinatoire

**Stratégie de sélection** :
- Intuition théorique (mécanismes complémentaires)
- Contraintes émergentes (CON-XXX suggèrent couplages)
- Économie de tests (priorité aux combinaisons prometteuses)

**Traçabilité** :

Chaque HYP[R1] doit documenter explicitement:
- Quelles HYP[R0] sont incluses (champ `parent_hypotheses`)
- Pourquoi cette combinaison spécifique
- Quelles HYP[R0] sont délibérément exclues

**Exemple** :
```
HYP-004[R1] {
  parent_hypotheses = [HYP-001, HYP-002]
  rationale = "Saturation + diffusion = régulation locale"
  excluded = [HYP-003]
  exclusion_reason = "Mécanisme orthogonal, testé séparément"
}
```

### 11.4 Pattern Γ non-markovien

**Γ avec mémoire ordre 1** :

Un Γ non-markovien stocke explicitement l'état précédent.

**Implémentation standard** :
```python
# operators/gamma_hyp_XXX.py

class NonMarkovianGamma:
    """Γ avec mémoire ordre 1 : Γ(C_t, C_t-1)"""
    
    def __init__(self, alpha=0.01):
        self.alpha = alpha
        self._previous_state = None  # Mémoire explicite
    
    def __call__(self, state: np.ndarray) -> np.ndarray:
        """Applique Γ(C_t, C_t-1)"""
        if self._previous_state is None:
            # Première itération : comportement markovien
            result = state.copy()
        else:
            # Itérations suivantes : utilise mémoire
            delta = state - self._previous_state
            result = state + self.alpha * delta
        
        # Stocke état actuel pour prochaine itération
        self._previous_state = state.copy()
        return result
```

**Usage dans TM** :
```python
gamma = NonMarkovianGamma(alpha=0.01)

for i, state in run_kernel(D, gamma, max_iterations=1000):
    # Le kernel appelle gamma(state)
    # Gamma accède automatiquement à son _previous_state interne
    pass
```

**Note importante** :
- Le **core** (`run_kernel`) reste aveugle
- C'est **Γ lui-même** qui gère sa mémoire
- Le kernel n'a pas besoin d'être modifié

## 14.1 PRINCIPE DE NON-RERUNS

### Règle absolue

**Toute modification de scoring, pondération, ou verdict doit être possible SANS réexécuter les runs du kernel.**

### Conséquences architecturales

1. **Données factuelles et analyses sont SÉPARÉES**
   - Les runs du kernel ne sont exécutés qu'une fois
   - Toute interprétation ultérieure lit les données existantes

2. **Pipeline est REJOUABLE**
   - Tests peuvent être ré-appliqués avec nouvelles implémentations
   - Scores peuvent être recalculés avec nouvelles pondérations
   - Verdicts peuvent être régénérés avec nouveaux seuils

3. **Configs sont EXTERNES**
   - Aucune règle de scoring hardcodée dans le code
   - Tous les paramètres dans des fichiers YAML versionnés

### Interdictions formelles

```
\INTERDICTIONS{
  - Forcer un rerun pour changer une règle de scoring
  - Mélanger données factuelles et interprétations dans une même table
  - Hardcoder des pondérations ou seuils dans le code
  - Créer un pipeline linéaire non rejouable
}
```

## 14.2 ARCHITECTURE BATCH_RUNNER

### Règle structurelle

**Un seul point d'entrée : `batch_runner.py`**

Le script unique gère trois modes d'exécution avec **chaînage automatique** des dépendances.

### Modes d'exécution

#### Mode `--brut`

**Responsabilité** : Collecte des données de runs

```bash
batch_runner --brut --gamma GAM-001
```

**Comportement** :
1. Exécute kernel pour tous (D, modifier, seed) applicables
2. Stocke dans `db_raw` :
   - Executions (métadonnées run)
   - Snapshots (états sauvegardés)
   - Metrics (métriques par itération)
3. **Aucun test appliqué**
4. **Aucune interprétation**

**Vérification préalable** :
- Vérifie si runs existent déjà dans `db_raw`
- Skip les runs existants (append-only)

---

#### Mode `--test`

**Responsabilité** : Application des tests sur données existantes

```bash
batch_runner --test --gamma GAM-001 --config default
```

**Comportement** :
1. **Vérifie que `db_raw` contient les runs nécessaires (cad les datas d'un run ayant mêmes id pour tous les paramètres)**
2. **Si runs manquants : exécute automatiquement `--brut` d'abord**
3. Lit les runs depuis `db_raw`
4. Applique tous les tests applicables (selon matrice d'applicabilité)
5. Calcule scores contextuels avec config spécifiée
6. Stocke dans `db_results` :
   - TestObservations (valeurs brutes tests)
   - TestScores (scores calculés avec config_id)

**Vérification préalable** :
- Si runs absents dans `db_raw` → exécute `--brut` automatiquement
- Vérifie si tests déjà appliqués pour cette config
- Skip les observations existantes (append-only par config_id)

**Rejouabilité** :
- Peut être relancé avec `--config alternative` sans refaire les runs
- Chaque config génère un nouveau jeu de scores avec son config_id

---

#### Mode `--verdict`

**Responsabilité** : Génération des verdicts agrégés

```bash
batch_runner --verdict --gamma GAM-001 --config default --thresholds strict
```

**Comportement** :
1. **Vérifie que `db_results` contient les scores pour config spécifiée**
2. **Si scores manquants : exécute automatiquement `--test` d'abord**
   - Qui lui-même exécutera `--brut` si nécessaire
3. Lit les scores depuis `db_results`
4. Calcule les 3 critères (Majorité, Robustesse, Score global)
5. Applique les seuils du fichier thresholds
6. Stocke dans `db_results` :
   - GammaVerdicts (verdict avec config_id + threshold_id)

**Vérification préalable** :
- Si scores absents pour config → exécute `--test` automatiquement
- Si runs absents → `--test` exécutera `--brut` automatiquement
- Cascade complète jusqu'au premier manquant

**Rejouabilité** :
- Peut être relancé avec `--thresholds alternative` sans refaire scoring
- Chaque combinaison (config, thresholds) génère un verdict distinct

---

### Mode `--all` (optionnel, raccourci)

**Responsabilité** : Chaîne complète en une commande

```bash
batch_runner --all --gamma GAM-001 --config default --thresholds strict
```

**Équivalent strict de** :
```bash
batch_runner --brut --gamma GAM-001
batch_runner --test --gamma GAM-001 --config default
batch_runner --verdict --gamma GAM-001 --config default --thresholds strict
```

**Usage** : Utile quand on sait que tout est stable. Sinon, modes séparés préférables.

---

### Chaînage automatique des dépendances

**Graphe de dépendances** :
```
--brut
  ↓ (vérifié automatiquement par --test)
--test (avec config X)
  ↓ (vérifié automatiquement par --verdict)
--verdict (avec config X, thresholds Y)
```

**Exemple concret** :
```bash
# Situation : db vides
batch_runner --verdict --gamma GAM-001 --config default --thresholds strict

# Exécution automatique :
# 1. Détecte absence runs dans db_raw
# 2. Exécute --brut d'abord
# 3. Détecte absence tests dans db_results
# 4. Exécute --test ensuite
# 5. Calcule verdict final
```

**Avantage** : Utilisateur n'a jamais besoin de savoir ce qui existe ou pas.

---

### Interdictions formelles

```
\INTERDICTIONS{
  - Créer batch_runner_raw.py, batch_runner_with_tests.py, compute_scores.py séparés
  - Forcer l'utilisateur à exécuter les modes dans le bon ordre
  - Manquer une vérification de dépendance (toujours vérifier existence)
  - Rendre --all obligatoire (modes séparés doivent fonctionner)
  - Créer des modes supplémentaires non documentés ici
}
```

## 14.3 SÉPARATION BASES DE DONNÉES

### Règle structurelle

**Deux bases séparées, rôles distincts et non interchangeables.**

### db_raw : Données factuelles

**Fichier** : `prc_database/prc_r0_raw.db`

**Contenu** :
```sql
Executions   -- Métadonnées des runs (gamma_id, params, d_base_id, seed, ...)
Snapshots    -- États sauvegardés (iteration, state_blob compressé)
Metrics      -- Métriques brutes par itération (norm, std, min, max, ...)
```

**Propriétés obligatoires** :
- **Append-only** : Jamais de UPDATE ou DELETE
- **Immuable** : Une fois écrit, jamais modifié
- **Complète** : Contient TOUTES les données nécessaires aux analyses futures

**Cas d'usage** :
- Mode `--brut` écrit exclusivement ici
- Modes `--test` et `--verdict` lisent exclusivement d'ici (sauf leurs propres résultats)

---

### db_results : Analyses et verdicts

**Fichier** : `prc_database/prc_r0_results.db`

**Contenu** :
```sql
TestObservations  -- Résultats bruts des tests (test_name, observation_data JSON)
TestScores        -- Scores calculés (config_id, test_name, score 0-1)
GammaVerdicts     -- Verdicts agrégés (config_id, threshold_id, verdict)
```

**Propriétés obligatoires** :
- **Rejouable** : Peut être supprimée et reconstruite depuis db_raw
- **Versionnée** : Chaque config/threshold génère un jeu de résultats distinct
- **Multiple** : Plusieurs verdicts coexistent pour les mêmes runs

**Cas d'usage** :
- Mode `--test` écrit TestObservations et TestScores
- Mode `--verdict` écrit GammaVerdicts
- Les trois modes peuvent lire d'ici pour vérifier existence
- On avertira un utilisateur voulant refaire un test déjà présent, prévenant que le nouveau résultat remplacera l'existant

---

### Relation entre les bases

**db_raw** est la **source de vérité unique**.

**db_results** est **dérivée** et peut contenir plusieurs interprétations des mêmes runs.

**Schéma de référence** :
```
db_raw.Executions (exec_id, gamma_id, params, d_base_id, seed, ...)
  ↓ (1-to-N)
db_results.TestObservations (exec_id, test_name, value, ...)
  ↓ (1-to-N)
db_results.TestScores (exec_id, test_name, config_id, score, ...)
  ↓ (agrégation)
db_results.GammaVerdicts (gamma_id, config_id, threshold_id, verdict, ...)
```

**Clé étrangère obligatoire** : `exec_id` dans `db_results` référence `db_raw.Executions.id`

---

### Interdictions formelles

```
\INTERDICTIONS{
  - Mélanger runs et verdicts dans une seule base
  - Stocker des interprétations dans db_raw
  - Modifier rétroactivement db_raw (sauf bugs critiques documentés)
  - Créer db_raw sans clé primaire stable (exec_id)
  - Créer db_results sans référence à exec_id
  - Stocker des observations sans config_id
}
```

## 14.4 TESTS ET SCORING

### Règle de séparation

**Tests observent, scoring interprète.**

Les tests retournent des **observations pures** (valeurs numériques + transitions).

Le scoring applique un **contexte** et des **pondérations** pour calculer des scores 0-1.

---

### Structure obligatoire des tests

**Tous les tests doivent retourner une observation** :

```python
@dataclass
class XxxObservation:
    """Observation du test XXX."""
    # Valeurs numériques brutes
    initial_value: float
    final_value: float
    min_value: float
    average_value: float
    max_value: float
    
    # Transitions qualitatives
    transition: str  # "preserved" | "created" | "destroyed" | "absent"
    
    # Métriques complémentaires
    trend: str  # "increasing" | "decreasing" | "stable" | "oscillating"
    
    # PAS de status PASS/FAIL
    # PAS de blocking boolean (c'est au scoring de décider)
```

**Exemples conformes** :
- `SymmetryObservation` (initial_symmetric, final_symmetric, transition, max_asymmetry)
- `NormObservation` (initial_norm, final_norm, evolution, max_norm)
- `DiversityObservation` (initial_diversity, final_diversity, ratio, evolution)

---

### Structure obligatoire du scoring contextuel

**Le scoring prend une observation et calcule un score 0-1** :

```python
def score_xxx(obs: XxxObservation, context: dict, weights: dict) -> float:
    """
    Calcule score contextuel 0-1.
    
    Args:
        obs: Observation brute du test
        context: Informations contextuelles (d_base_id, gamma_id, ...)
        weights: Pondérations depuis config YAML
    
    Returns:
        Score entre 0.0 et 1.0
    """
    # Échelle 0-1 selon transition et contexte
    # 0.0 = Destruction/perte
    # 0.5 = Neutre/conservation
    # 1.0 = Création/amélioration
    
    # Exemple symétrie
    if obs.transition == "destroyed":
        score_raw = 0.0
    elif obs.transition in ["preserved", "absent"]:
        score_raw = 0.5
    elif obs.transition == "created":
        score_raw = 1.0
    
    # Pondération depuis config
    weight = weights.get(test_name, 1.0)
    return score_raw * weight
```

---

### Échelle de scoring universelle

**Tous les tests utilisent l'échelle 0-1** :

```
PROPRIÉTÉS CONSERVABLES (symétrie, norme, diversité, positivité)
├─ 0.0 = Destruction/perte complète
├─ 0.5 = Neutre/conservation
└─ 1.0 = Création/amélioration

MÉTRIQUES CONTINUES (vitesse convergence, rayon spectral)
├─ Normalisation selon distribution observée
├─ 0.0 = borne inférieure du critère
├─ 0.5 = Médiane
└─ 1.0 = borne supérieure du critère

\NOTE{les bornes seront définies par test, dans certains cas un convergence rapide permet d'évaluer la stabilité sous pertubation, dans d'autres processes, une convergence lente obtiendra un meilleur score}
```

**Exemple concret symétrie** :
```
(initial, final) → score
(0, 0) → 0.5  # Absent → Absent = Neutre
(1, 0) → 0.0  # Présent → Absent = Destruction
(0, 1) → 1.0  # Absent → Présent = Création
(1, 1) → 0.5  # Présent → Présent = Conservation (neutre)
```

---

### Applicabilité technique

**Matrice d'applicabilité obligatoire** : `tests/utilities/applicability.py`

**Contenu** :
```python
TEST_APPLICABILITY = {
    "test_symmetry_preservation": {
        "requires_rank": 2,
        "requires_square": True,
        "d_types": ["SYM", "ASY"],  # Pas R3
    },
    "test_spectral_evolution": {
        "requires_rank": 2,
        "requires_square": True,
        "d_types": ["SYM", "ASY"],  # Applicable aux deux
    },
    "test_norm_evolution": {
        "requires_rank": None,      # Tous rangs
        "d_types": ["SYM", "ASY", "R3"],  # Tous types
    },
    # ...
}

def is_test_applicable(test_name: str, d_base_id: str, 
                       state_shape: tuple) -> bool:
    """Vérifie compatibilité test/contexte."""
    # ...
```

**Règle** : Avant d'appliquer un test, vérifier applicabilité. Si non applicable, stocker `applicable=False` dans TestObservations.

---

### Interdictions formelles

```
\INTERDICTIONS{
  - Tests retournent status="PASS"/"FAIL"
  - Tests retournent blocking=True/False (c'est au scoring de décider)
  - Verdicts hardcodés dans les tests
  - Scoring non rejouable (pondérations hardcodées)
  - Appliquer un test sans vérifier applicabilité
  - Échelles de scoring incohérentes entre tests
}

## 14.5 CONFIGS EXTERNES

### Règle de configuration

**Toute règle de scoring ou verdict est dans un fichier YAML versionné.**

Aucune pondération, seuil, ou règle de décision ne doit être hardcodée dans le code.

---

### Structure des configs de pondération

**Fichier** : `config/weights_<version>.yaml`

**Format obligatoire** :
```yaml
version: "1.0"
description: "Pondérations par défaut selon Charter 5.1"

weights:
  # Tests universels (critiques)
  UNIV-001: 2.0  # Norme
  UNIV-002: 2.0  # Diversité
  UNIV-003: 2.0  # Convergence
  
  # Tests structurels (importants)
  SYM-001: 1.5   # Préservation symétrie
  STR-001: 1.5   # Rang
  
  # Tests bornes (secondaires)
  BND-001: 1.0   # Respect bornes
  LOC-001: 1.0   # Propagation
```

**Usage** :
```bash
batch_runner --test --config weights_default
batch_runner --test --config weights_conservative
```

Chaque config génère un jeu de scores distinct avec son `config_id`.

---

### Structure des configs de seuils

**Fichier** : `config/thresholds_<version>.yaml`

**Format obligatoire** :
```yaml
version: "1.0"
description: "Seuils selon Feuille de Route Section 7"

# Critères SURVIVES[R0] (OU logique)
survives:
  score_min: 12.0        # Sur 20
  robustness_min: 60.0   # Pourcentage
  majority_min: 50.0     # Pourcentage

# Critères FLAGGED_FOR_REVIEW (ET logique)
flagged:
  score_max: 8.0
  robustness_max: 40.0
  majority_max: 20.0

# Entre les deux : WIP[R0-closed]
```

**Usage** :
```bash
batch_runner --verdict --thresholds thresholds_default
batch_runner --verdict --thresholds thresholds_strict
```

Chaque combinaison (config_id, threshold_id) génère un verdict distinct.

---

### Versionnement obligatoire

**Toute config doit avoir** :
- Un identifiant unique (ex: `default`, `v1_conservative`, `phase1_strict`)
- Une version explicite (ex: `"1.0"`)
- Une description claire

**Stockage dans DB** :
```sql
TestScores (
  exec_id,
  test_name,
  config_id TEXT,  -- "weights_default"
  score REAL,
  ...
)

GammaVerdicts (
  gamma_id,
  config_id TEXT,      -- "weights_default"
  threshold_id TEXT,   -- "thresholds_strict"
  verdict TEXT,
  ...
)
```

---

### Interdictions formelles

```
\INTERDICTIONS{
  - Pondérations hardcodées dans le code Python
  - Seuils de verdict non paramétrables
  - Configs non versionnées (pas d'identifiant unique)
  - Modifier une config existante (créer une nouvelle version)
  - Stocker des scores sans config_id
  - Calculer des verdicts sans référence à threshold_id
}
```

## 14.6 VERDICTS MULTIPLES

### Règle de multiplicité

**Plusieurs verdicts peuvent coexister pour les mêmes données de runs.**

Chaque verdict est **identifié de manière unique** par :
- `gamma_id` (quelle hypothèse)
- `config_id` (quelle pondération)
- `threshold_id` (quels seuils)

---

### Structure table GammaVerdicts

**Schéma obligatoire** :
```sql
CREATE TABLE GammaVerdicts (
  id INTEGER PRIMARY KEY,
  gamma_id TEXT NOT NULL,
  config_id TEXT NOT NULL,        -- Référence config/weights_XXX.yaml
  threshold_id TEXT NOT NULL,     -- Référence config/thresholds_XXX.yaml
  
  -- 3 critères
  majority_pct REAL,              -- % configs PASS
  robustness_pct REAL,            -- % D avec ≥1 config viable
  score_global REAL,              -- Moyenne pondérée
  
  -- Verdict
  verdict TEXT NOT NULL,          -- "SURVIVES[R0]" | "WIP[R0-closed]" | "FLAGGED_FOR_REVIEW"
  verdict_reason TEXT,
  
  -- Traçabilité
  computed_at TEXT,
  
  UNIQUE(gamma_id, config_id, threshold_id)
);
```

---

### Calcul des 3 critères

**Majorité** :
```
majority_pct = (Nombre de configs PASS) / (Nombre total de configs) × 100
```

Où une config = (gamma_params, d_base_id, modifier_id, seed)

**Robustesse** :
```
robustness_pct = (Nombre de D avec ≥1 config viable) / (Nombre total D testés) × 100
```

**Score global** :
```
score_global = median(scores de toutes configs) × 20 / max_weight
```

Pour ramener sur échelle /20.

---

### Règles de verdict

**SURVIVES[R0]** (OU logique) :
```
Si score_global ≥ score_min
OU robustness_pct ≥ robustness_min
OU majority_pct ≥ majority_min
→ SURVIVES[R0]
```

**FLAGGED_FOR_REVIEW** (ET logique) :
```
Si score_global < score_max
ET robustness_pct < robustness_max
ET majority_pct < majority_max
→ FLAGGED_FOR_REVIEW
```

**WIP[R0-closed]** (par défaut) :
```
Tout ce qui n'est ni SURVIVES ni FLAGGED
→ WIP[R0-closed]
```

**Note critique** : Aucun REJECTED[R0] automatique. Les FLAGGED_FOR_REVIEW nécessitent une review manuelle pour devenir REJECTED[R0].

---

### Exemple de verdicts multiples

**Mêmes runs, configs différentes** :

```sql
-- Config conservative
GammaVerdicts (
  gamma_id='GAM-005',
  config_id='weights_conservative',
  threshold_id='thresholds_default',
  verdict='WIP[R0-closed]'
)

-- Config lenient
GammaVerdicts (
  gamma_id='GAM-005',
  config_id='weights_default',
  threshold_id='thresholds_lenient',
  verdict='SURVIVES[R0]'
)
```

→ Permet de comparer l'impact des configs sans refaire les runs.

---

### Interdictions formelles

```
\INTERDICTIONS{
  - Verdict unique et définitif sans config_id/threshold_id
  - Écraser un verdict existant (INSERT uniquement)
  - DELETE sur GammaVerdicts (append-only)
  - Calculer verdict sans les 3 critères
  - REJECTED[R0] automatique (toujours manuel après FLAGGED)
}
```

## 14.7 VALIDATION ET COHÉRENCE

### Règle de validation

**Chaque mode du batch_runner doit inclure des validations** pour vérifier la cohérence des données avant/après exécution.

Ces validations sont **séparées de Section 14** mais **obligatoires** dans l'implémentation.

---

### Validations obligatoires par mode

#### Mode `--brut`
- Vérifier que tous les runs sont bien stockés dans `db_raw.Executions`
- Vérifier que snapshots correspondent aux itérations enregistrées
- Vérifier que metrics sont cohérentes (pas de NaN/Inf inattendus)

#### Mode `--test`
- Vérifier que tous les runs nécessaires existent dans `db_raw`
- Vérifier que tous les tests applicables ont été appliqués
- Vérifier que TestObservations contient des données cohérentes
- Vérifier que TestScores sont calculés pour la config spécifiée

#### Mode `--verdict`
- Vérifier que TestScores existent pour la config spécifiée
- Vérifier que les 3 critères sont calculés
- Vérifier que le verdict est cohérent avec les seuils
- Vérifier que GammaVerdicts contient config_id et threshold_id

---

### Tests de cohérence inter-bases

**Intégrité référentielle** :
- Tous les `exec_id` dans `db_results` doivent exister dans `db_raw.Executions`
- Tous les `test_name` doivent être dans la liste `tests/utilities/__all__`
- Tous les `config_id` doivent correspondre à des fichiers YAML existants

**Complétude** :
- Pour chaque run dans `db_raw`, vérifier qu'au moins un test a été appliqué
- Pour chaque gamma testée, vérifier qu'au moins un verdict existe

**Ces validations ne sont PAS documentées dans Section 14** pour ne pas alourdir.

Elles seront implémentées dans :
- `tests/validation/validate_db_raw.py`

Et exécutées via :
```bash
python tests/validation/run_all_validations.py
```

---

### Interdictions formelles

```
\INTERDICTIONS{
  - Accepter des données incohérentes silencieusement
  - Continuer le pipeline si une validation échoue
  - Stocker des résultats sans vérifier les dépendances
}

## 14.8.1 PRINCIPE FONDAMENTAL

**RÈGLE ABSOLUE** : Les tests OBSERVENT, le scoring INTERPRÈTE.

```
\SEPARATION_STRICTE{
  Tests (L3)    : Mesures factuelles → dict de valeurs numériques
  Scoring (L3)  : Interprétation contextuelles → scores 0-1
  Verdicts (L3) : Agrégation scores → décisions finales
}
```

**Interdiction formelle** :
- ❌ Tests ne retournent JAMAIS "PASS" ou "FAIL" (c'est une interprétation)
- ❌ Tests ne jugent JAMAIS si une observation est "bonne" ou "mauvaise"
- ❌ Tests ne contiennent JAMAIS de logique de scoring
- ❌ Utilisation d'objets ou classes pour stocker observations

**Autorisations** :
- ✅ Tests retournent "ERROR", "NOT_APPLICABLE" (status technique, pas verdict)
- ✅ Tests retournent valeurs numériques brutes
- ✅ Tests décrivent transitions factuelles ("increasing", "preserved", etc.)
- ✅ Format dict pur, JSON-sérialisable

---

## 14.8.2 FORMAT DICT STANDARDISÉ

**Tout test DOIT retourner un dict avec cette structure exacte** :

```python
{
    # =========================================================================
    # IDENTIFICATION (OBLIGATOIRE)
    # =========================================================================
    'test_name': str,        # ID du test (ex: "UNIV-001")
    
    # =========================================================================
    # STATUS TECHNIQUE (OBLIGATOIRE)
    # =========================================================================
    'status': str,           # Valeurs autorisées :
                             #   "SUCCESS"          - Test exécuté avec succès
                             #   "ERROR"            - Erreur technique (NaN, exception)
                             #   "NOT_APPLICABLE"   - Test non applicable au contexte
    
    'message': str,          # Description factuelle (pas de jugement)
                             # OK : "Norme finale = 15.3"
                             # ❌ : "Norme trop élevée, échec"
    
    # =========================================================================
    # VALEURS POUR SQL (OBLIGATOIRE)
    # =========================================================================
    'initial_value': float,  # Mesure à t=0
    'final_value': float,    # Mesure à t=T
    'transition': str,       # Description factuelle transition
                             # Ex: "stable", "increasing", "decreasing"
                             #     "preserved", "created", "destroyed"
                             # PAS : "acceptable", "problematic", "failed"
    
    # =========================================================================
    # SCORE BRUT (OBLIGATOIRE)
    # =========================================================================
    'value': float,          # Valeur principale pour calculs scoring
                             # Ex: norme finale, ratio, distance, etc.
    
    # =========================================================================
    # MÉTADONNÉES (OPTIONNEL)
    # =========================================================================
    'metadata': dict         # Détails additionnels JSON-sérialisables
                             # Ex: {'max': 18.2, 'min': 0.3, 'trend': -0.02}
}
```

### Phase 2 : Scoring (interprétation contextuelle)

```python
def score_observation(test_name, observation, context, weights):
    """
    Interprète l'observation selon le contexte.
    
    C'est ICI qu'on décide si c'est bon ou mauvais.
    """
    transition = observation['transition']
    
    # ✅ CORRECT : Interprétation contextuelle
    if transition == 'explosive':
        score = 0.0  # Jugement : explosion = mauvais
    elif transition == 'stable':
        score = 1.0  # Jugement : stabilité = bon
    else:
        score = 0.5
    
    return {'score': score, 'weight': weights[test_name], ...}
```

## 14.8.5 INTERDICTION CLASSES ET OBJETS

**Règle absolue** : Aucune classe ou objet pour stocker observations.

### ❌ Interdit

```python
# NE JAMAIS FAIRE
@dataclass
class NormResult:
    initial_norm: float
    final_norm: float
    status: str

# NE JAMAIS FAIRE
class TestResult:
    def __init__(self, data):
        self.status = data['status']
        ...
```

### ✅ Autorisé

```python
# TOUJOURS FAIRE : dict pur
def run_test(history, context):
    return {
        'test_name': 'UNIV-001',
        'status': 'SUCCESS',
        'initial_value': 10.5,
        ...
    }
```

### Raison

1. **Sérialisabilité** : dict → JSON direct, pas besoin de serializers custom
2. **Simplicité** : Pas de classes à maintenir, pas de méthodes à documenter
3. **Compatibilité** : SQLite attend dict, pas d'objets
4. **Évolutivité** : Ajouter un champ = ajouter une clé, pas modifier une classe

## 14.8.7 TEMPLATE TEST CONFORME

```python
# tests/test_xxx.py
"""
Test XXX : [Description]

Applicable : [Conditions]
Mesure : [Formule]
"""

import numpy as np

TEST_ID = "XXX-001"
REQUIRES_RANK = None  # ou 2, 3
D_TYPES = ["SYM", "ASY", "R3"]
WEIGHT_DEFAULT = 1.0


def _safe_compute(state):
    """
    Helper avec gestion NaN/Inf.
    Retourne float ou None si erreur.
    """
    if np.any(np.isnan(state)) or np.any(np.isinf(state)):
        return None
    
    # Calcul sécurisé
    return float(np.linalg.norm(state))


def run_test(history, context):
    """
    Mesure [propriété factuelle].
    
    Args:
        history: Liste états [D_0, ..., D_T]
        context: dict {d_base_id, gamma_id, seed, modifier_id}
    
    Returns:
        dict standardisé conforme Section 14.8.2
    """
    # Vérification base
    if not history or len(history) < 2:
        return {
            'test_name': TEST_ID,
            'status': 'ERROR',
            'message': 'Historique insuffisant',
            'initial_value': 0.0,
            'final_value': 0.0,
            'transition': 'error',
            'value': 0.0,
            'metadata': {}
        }
    
    # Vérifier applicabilité
    if context['d_base_id'].startswith('R3') and REQUIRES_RANK == 2:
        return {
            'test_name': TEST_ID,
            'status': 'NOT_APPLICABLE',
            'message': 'Test nécessite rang 2',
            'initial_value': 0.0,
            'final_value': 0.0,
            'transition': 'not_applicable',
            'value': 0.0,
            'metadata': {}
        }
    
    # Calcul observation
    values = [_safe_compute(state) for state in history]
    
    # Gestion erreurs numériques
    if any(v is None for v in values):
        first_error = next(i for i, v in enumerate(values) if v is None)
        return {
            'test_name': TEST_ID,
            'status': 'ERROR',
            'message': f'NaN/Inf détecté à itération {first_error}',
            'initial_value': values[0] if values[0] else 0.0,
            'final_value': 0.0,
            'transition': 'nan_detected',
            'value': 0.0,
            'metadata': {'first_error_iteration': first_error}
        }
    
    # Calcul transition (description factuelle)
    initial = values[0]
    final = values[-1]
    ratio = final / (initial + 1e-10)
    
    if abs(ratio - 1.0) < 0.1:
        transition = "stable"
    elif ratio > 1.0:
        transition = "increasing"
    else:
        transition = "decreasing"
    
    # Retour conforme
    return {
        'test_name': TEST_ID,
        'status': 'SUCCESS',
        'message': f'Mesure: {initial:.2f} → {final:.2f}',
        'initial_value': initial,
        'final_value': final,
        'transition': transition,
        'value': final,
        'metadata': {
            'max': max(values),
            'min': min(values),
            'ratio': ratio,
            'values': values  # Historique complet si nécessaire
        }
    }
```
