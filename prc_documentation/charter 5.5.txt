## SECTION 1-3 : FONDATIONS

### Axiomes (A1-A3)
- A1: Dissymétrie informationnelle D irréductible
- A2: Mécanisme Γ agissant sur D
- A3: Aucun Γ stable ne peut annuler D complètement

### Hiérarchie niveaux
```
L0=ontological → L1=epistemic → L2=theoretical → L3=operative → L4=documentary
Règle : L(n) ne référence que L(≤n-1)
```

### Core = 2 fonctions aveugles
```python
prepare_state(base, modifiers) → np.ndarray
run_kernel(state, gamma, ...) → Generator[iteration, state]
```

**Règles K1-K5** : Core aveugle (pas de validation, pas de sémantique, pas de classes)

## SECTION 6 : PIPELINE OPÉRATIONNEL

### Workflow
```
1. Formulation HYP → 2. Spécification D/Γ → 3. Création TM
→ 4. Exécution kernel (db_raw) → 5. Tests observations (db_results)
→ 6. Verdicts patterns (rapports) → 7. Discussion → 8. Mise à jour STATUS
→ 9. CON si pertinent → 10. Nouvelle HYP[R1] si pertinent
```

### Architecture batch_runner

**Principe** : 1 point d'entrée, 3 modes, chaînage auto

```bash
# Mode 1 : Collecte données
batch_runner --brut --gamma GAM-001
# → Exécute kernel, stocke db_raw (append-only, immuable)

# Mode 2 : Application tests
batch_runner --test --gamma GAM-001 --params default_v1
# → Vérifie db_raw, auto-lance --brut si manquant
# → Applique tests, stocke observations db_results

# Mode 3 : Génération verdicts
batch_runner --verdict --params default_v1 --verdict default_v1
# → Vérifie db_results, auto-lance --test si manquant
# → Analyse patterns, génère rapports fichiers

# Raccourci
batch_runner --all --gamma GAM-001 --params default_v1 --verdict default_v1
```

**Règle non-reruns** : Toute modification verdict SANS réexécuter kernel/tests

### Bases de données

**db_raw** (immuable, append-only) :
```sql
Executions   -- Métadonnées runs
Snapshots    -- États sauvegardés (compressed)
Metrics      -- Métriques brutes par iteration
```

**db_results** (rejouable, versionné) :
```sql
TestObservations  -- Résultats tests (dict v2)
```

**Aucune table** : ConfigRegistry, TestScores, GammaVerdicts (supprimés)

## SECTION 8 : ARCHITECTURE REGISTRES

### Principe

**R8-A** : Registres = modules contenant fonctions mathématiques réutilisables.

**R8-B** : Toute fonction registre produit UNE OBSERVATION (float), pas un score.

### BaseRegistry

```python
class BaseRegistry(ABC):
    REGISTRY_KEY: str = None  # Unique
    
    def __init__(self):
        self._functions: Dict[str, Callable] = {}
        self._register_all_functions()
    
    def get_function(self, registry_key: str) -> Callable:
        _, function_name = registry_key.split('.', 1)
        return self._functions[function_name]
```

### Décorateur

```python
@register_function("matrix_norm")
def compute_norm(self, state: np.ndarray, norm_type='fro') -> float:
    """
    Calcule norme tenseur.
    
    Args:
        state: Tenseur rang N
        norm_type: Type norme
    
    Returns:
        float: Valeur norme
    
    Notes:
        - Frobenius : O(n²)
        - Spectral : matrice 2D uniquement
    """
    return float(np.linalg.norm(state, norm_type))
```

### Règles création registre

**R8-C** : Règles strictes :
1. Hériter de BaseRegistry
2. REGISTRY_KEY unique, lowercase
3. Décorateur @register_function obligatoire
4. Signature: `func(self, state, **params) -> float`
5. `state` toujours premier paramètre
6. Tous paramètres après `state` ont valeurs par défaut
7. Retourner float
8. Docstring complète (Args, Returns, Raises, Notes)
9. Gestion erreurs explicite
10. Fonction pure (pas d'effets de bord)

### Registres disponibles

| Registre | Fonctions | Description |
|----------|-----------|-------------|
| algebra | 8 | Normes, traces, déterminants |
| spectral | 6 | FFT, valeurs propres |
| statistical | 7 | Entropie, moments |
| spatial | 6 | Gradients, laplacien |
| pattern | 6 | Périodicité, clustering |
| topological | 6 | Composantes, Euler |
| graph | 6 | Densité, centralité |

## SECTION 14 : NORMALISATION PATTERNS

### Principe

**R14-A** : VerdictEngine normalise observations **à la volée**. Normalisation **non stockée**.

**R14-B** : Normalisation sur **distributions locales** (ensemble runs pour métrique donnée).

**R14-C** : Objectif : détecter **corrélations et ruptures**, pas comparer amplitudes.

### Méthode par défaut

**R14-D** : Méthode default : **Robust scaling**

```
normalized = (x - median) / IQR
```

- `median` = médiane échantillon
- `IQR` = Q3 - Q1 (écart interquartile)

**R14-E** : Justification : insensible valeurs extrêmes (explosions/collapses recherchés).

**R14-F** : Méthodes alternatives :
- `z_score` : (x - mean) / std
- `percentile` : percentileofscore(values, x) / 100

### Configuration

**R14-G** : Méthode configurable par pattern :

```yaml
normalization:
  default_method: "robust"
  scope: "per_test"
  outlier_threshold: 3.0
  
  per_pattern:
    non_discriminant: "percentile"
    seed_unstable: "robust"
```

**R14-H** : Méthode = **partie intégrante définition pattern**. Pattern = (projection + normalisation + critère).

### Seuils normalisés

**R14-I** : Seuils opèrent dans **espace normalisé**, pas métrique brut.

**R14-J** : Terminologie neutre : `normalized_threshold` (pas `z_threshold`).

```yaml
patterns:
  non_discriminant:
    method: "robust"
    normalized_threshold: -2.0
    min_observations: 10
```

## SECTION 15 : PATTERNS - DÉTECTION

### Architecture

**R15-A** : VerdictEngine charge **toutes observations** :

```python
observations = load_all_observations(params_config_id)
```

**R15-B** : Patterns détectés sur observations **brutes + normalisées**.

**R15-C** : Aucun stockage intermédiaire. Rapports générés directement.

### Types patterns

**R15-D** : 7 types détectés :

1. **NON_DISCRIMINANT** : Toutes observations normalisées < threshold
2. **OVER_DISCRIMINANT** : Toutes observations normalisées > threshold
3. **D_CORRELATED** : Variance inter-d_encoding_id > threshold
4. **MODIFIER_CORRELATED** : Variance inter-modifier_id > threshold
5. **SEED_UNSTABLE** : Variance inter-seeds > threshold (à D/modifier fixés)
6. **SYSTEMATIC_ANOMALY** : Fraction observations |normalized| > threshold dépasse ratio
7. **CONTEXTUAL_BEHAVIOR** : Combinaison D_CORRELATED ou MODIFIER_CORRELATED

### Algorithmes

**NON_DISCRIMINANT** :
```python
normalized = normalize(values, method="robust")
if all(n < normalized_threshold for n in normalized):
    pattern = "NON_DISCRIMINANT"
```

**D_CORRELATED** :
```python
by_d = group_by(observations, 'd_encoding_id')
d_means_normalized = [normalize(group).mean() for group in by_d.values()]
variance = np.var(d_means_normalized)
if variance > variance_threshold:
    pattern = "D_CORRELATED"
```

**SYSTEMATIC_ANOMALY** :
```python
normalized = normalize(values, method="robust")
extreme_count = sum(1 for n in normalized if abs(n) > normalized_threshold)
if extreme_count / len(normalized) > ratio_threshold:
    pattern = "SYSTEMATIC_ANOMALY"
```

## SECTION 19 : INTERDICTIONS CRITIQUES

### Scoring

**R19-A** : Formellement interdit :
- Réintroduire table TestScores
- Créer couche normalisation globale préalable
- Stocker scores intermédiaires
- Utiliser vocabulaire "pathology_score", "test_score"

### Téléologie

**R19-B** : TEST_WEIGHT ne doit jamais être :
- Modifiable par config externe
- Utilisé pour filtrer/prioriser exécution
- Interprété comme "qualité" test

### Métriques

**R19-C** : Toute métrique `COMPUTATION_SPECS` **doit** être analysable par patterns.

**R19-D** : Aucun skip silencieux autorisé.

### Core

**R19-E** : Core reste aveugle :
- Aucune validation contenu (symétrie, bornes)
- Aucune classe State, Operator
- Aucun branchement dépendant de D ou Γ

### Tests

**R19-F** : Tests ne retournent jamais :
- "PASS"/"FAIL"
- Classes/objets
- Paramètres hardcodés
- Jugements ("bon", "mauvais")

### Pipeline

**R19-G** : Pipeline strictement séquentiel :
- Aucun reruns pour changer verdict
- Aucun mélange db_raw et db_results
- Aucune modification rétroactive db_raw

### Vocabulaire

**R19-H** : Nomenclature stricte :
- ❌ "matrice de corrélation" → ✅ "tenseur rang 2"
- ❌ "graphe" → ✅ "patterns dans tenseur"
- ❌ "position (i,j)" → ✅ "indice (i,j)"
- ❌ "d_base_id" → ✅ "d_encoding_id"

## SECTION 20 : GLOSSAIRE

| Terme | Définition | Exemple |
|-------|------------|---------|
| **Registre** | Module fonctions réutilisables | `algebra_registry.py` |
| **registry_key** | Identifiant unique fonction | `"algebra.matrix_norm"` |
| **run_metadata** | Métadonnées exécution | `{gamma_id, d_encoding_id, ...}` |
| **history** | Liste snapshots (~200) | `[state_0, ..., state_199]` |

| **post_processor** | Transformation post-calcul | `"round_4"` |
| **COMPUTATION_SPECS** | Spécifications métriques | `{'metric': {'registry_key': ...}}` |
| **APPLICABILITY_SPEC** | Contraintes techniques | `{'requires_rank': 2}` |
| **observation** | Dict retour test | `{'statistics': {...}, 'evolution': {...}}` |
| **pattern** | Régularité détectée cross-runs | `"D_CORRELATED"` |
| **normalized** | Valeur dans espace normalisé | `(x - median) / IQR` |
